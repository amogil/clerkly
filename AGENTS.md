# Руководство для Агентов

## Определения и Термины

- **Максимальное покрытие кода**: Минимум 85% покрытия строк кода тестами, стремление к 90%+
- **Краевые случаи**: Тестирование граничных значений, пустых данных, null/undefined, максимальных/минимальных значений
- **Исключительные ситуации**: Тестирование ошибок сети, недоступности ресурсов, таймаутов, некорректных входных данных
- **Компонент**: Логическая единица системы (класс, модуль, функция, UI-компонент)

## Правила Именования Компонентов

При написании спецификаций и документации:

- **Названия компонентов** должны быть написаны на английском языке
- **Не используйте подчеркивания** в названиях компонентов
- **Можно использовать несколько слов** для описания компонента
- **При ссылке на компоненты** всегда заключайте их в кавычки

### Примеры правильного именования:

✅ **Правильно:**

- "Main Process" (вместо Главный_Процесс)
- "OAuth Flow" (вместо Поток_OAuth)
- "Sidebar State" (вместо Состояние_Сайдбара)
- "Database Connection" (вместо Соединение*С*Базой_Данных)
- "Logo Component" (вместо Компонент_Логотипа)

❌ **Неправильно:**

- Главный_Процесс
- OAuth_Flow
- Sidebar_State
- Database_Connection
- Logo_Component

### Использование в тексте:

```markdown
КОГДА "Main Process" инициализируется, ТО "Renderer Process" ДОЛЖЕН запуститься
ЕСЛИ "OAuth Flow" завершается успешно, ТО "User Session" ДОЛЖНА быть создана
```

## Обязательная Валидация Изменений

**КРИТИЧЕСКИ ВАЖНО**: Каждое изменение в коде, спецификациях или тестах ДОЛЖНО быть провалидировано перед завершением работы.

### 1. Валидация Спецификаций

Все спецификации ДОЛЖНЫ быть:

- **Полными**: Покрывать все аспекты функциональности
- **Непротиворечивыми**: Не содержать конфликтующих требований
- **Не избыточными**: Не дублировать требования между спецификациями

**Действия при нарушении:**

- Исправить спецификации самостоятельно, если проблема очевидна
- Уточнить у пользователя при неясностях или конфликтах требований

### 2. Валидация Тестов

#### Модульные тесты:

```bash
npm test
```

#### Функциональные тесты:

```bash
npm run test:functional
```

**Требования:**

- Все тесты ДОЛЖНЫ проходить без ошибок
- При падении тестов: исправить код или обновить тесты
- Новая функциональность ДОЛЖНА быть покрыта тестами
- **МАКСИМАЛЬНОЕ ПОКРЫТИЕ**: Код и требования ДОЛЖНЫ быть максимально покрыты как модульными, так и функциональными тестами
- **КРАЕВЫЕ СЛУЧАИ**: Особое внимание ДОЛЖНО уделяться тестированию краевых случаев, граничных значений и исключительных ситуаций

### 3. Валидация Линтеров

#### ESLint (проверка качества кода):

```bash
npm run lint
```

#### Автоматическое исправление ESLint:

```bash
npm run lint:fix
```

#### Prettier (форматирование кода):

```bash
npm run format:check
```

#### Автоматическое форматирование:

```bash
npm run format
```

**Требования:**

- Все линтеры ДОЛЖНЫ проходить без замечаний
- Использовать автоматическое исправление где возможно
- Ручные исправления для сложных случаев

### 4. Дополнительные Валидации

#### TypeScript компиляция:

```bash
npm run build
```

#### Проверка зависимостей:

```bash
npm audit
```

**Примечание**: Уязвимости в dev-зависимостях (tar, vite) не критичны для production, но должны отслеживаться. Обновлять только при необходимости, чтобы не нарушить совместимость.

#### Проверка покрытия тестами:

```bash
# Покрытие проверяется через coverage.test.ts
npm test -- tests/requirements/coverage.test.ts
```

#### Валидация требований:

- Проверить корректность ссылок на требования в тестах
- Убедиться что все требования покрыты тестами
- Проверить соответствие кода спецификациям

### 5. Правила Выполнения Команд Валидации

**ВАЖНО**: При валидации НЕ объединяйте команды через `&&` или другие операторы. Выполняйте каждую команду отдельно.

❌ **Неправильно:**

```bash
npm run build && npm run lint && npm run format:check
npm test && npm run test:functional
```

✅ **Правильно:**

```bash
npm run build
npm run lint
npm run format:check
npm test
npm run test:functional
```

**Причины:**

- Лучшая диагностика ошибок
- Возможность исправления проблем поэтапно
- Избежание прерывания всей цепочки при единичной ошибке

### 6. Порядок Валидации

1. **Спецификации** → исправить/уточнить
2. **TypeScript компиляция** → исправить ошибки
3. **Линтеры** → исправить автоматически/вручную
4. **Модульные тесты** → исправить код/тесты
5. **Функциональные тесты** → исправить код/тесты
6. **Покрытие тестами** → добавить недостающие тесты
7. **Финальная проверка** → убедиться что все работает

### 7. Критерии Завершения

Работа считается завершенной ТОЛЬКО когда:

- ✅ Все спецификации корректны и согласованы
- ✅ TypeScript компилируется без ошибок
- ✅ ESLint проходит без замечаний
- ✅ Prettier форматирование корректно
- ✅ Все модульные тесты проходят
- ✅ Все функциональные тесты проходят
- ✅ Покрытие тестами соответствует требованиям
- ✅ Все требования покрыты тестами
- ✅ **Максимальное покрытие кода** достигнуто (минимум 85%, стремиться к 90%+)
- ✅ **Краевые случаи протестированы**: граничные значения, исключения, ошибочные состояния
- ✅ **Исключительные ситуации покрыты**: сетевые ошибки, недоступность ресурсов, некорректные данные
