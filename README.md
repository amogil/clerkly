# Clerkly - AI Agent для менеджеров

Clerkly - это Electron-приложение для Mac OS X, представляющее собой AI-ассистента для менеджеров. На текущем этапе реализуется базовая платформа приложения с локальным хранением данных, нативным Mac OS X интерфейсом и комплексным тестовым покрытием.

## Требования к системе

- **Node.js**: 18.0.0 или выше (рекомендуется 18.x или 20.x LTS)
- **Mac OS X**: 10.13 (High Sierra) или выше
- **npm**: 8.0.0 или выше (устанавливается вместе с Node.js)
- **Свободное место на диске**: минимум 500 MB для установки зависимостей и сборки

```bash
# Клонирование репозитория (если еще не клонирован)
```

**Примечание**: Первая установка может занять несколько минут, так как устанавливаются нативные модули (better-sqlite3) и Electron.

### Настройка Google OAuth

Для работы авторизации через Google необходимо настроить OAuth 2.0:

1. **Создайте проект в Google Cloud Console**:
   - Перейдите на [Google Cloud Console](https://console.cloud.google.com/)
   - Создайте новый проект или выберите существующий
   - Включите Google+ API для проекта

2. **Настройте OAuth consent screen**:
   - Перейдите в "APIs & Services" → "OAuth consent screen"
   - Выберите "External" user type
   - Заполните обязательные поля (название приложения, email поддержки)
   - Добавьте scopes: `openid`, `email`, `profile`

3. **Создайте OAuth 2.0 Client ID**:
   - Перейдите в "APIs & Services" → "Credentials"
   - Нажмите "Create Credentials" → "OAuth client ID"
   - Выберите "Desktop app" как тип приложения
   - Скопируйте Client ID

4. **Настройте Client ID в коде**:
   - Откройте файл `src/main/auth/OAuthConfig.ts`
   - Замените `'YOUR_GOOGLE_CLIENT_ID_HERE'` на ваш реальный Client ID:
   ```typescript
   export const OAUTH_CONFIG = {
     clientId: 'your-client-id.apps.googleusercontent.com', // Ваш Client ID
     redirectUri: 'clerkly://oauth/callback',
     scopes: ['openid', 'email', 'profile'],
   } as const;
   ```

5. **Настройте Redirect URI**:
   - В Google Cloud Console добавьте `clerkly://oauth/callback` в список разрешенных Redirect URIs
   - Это необходимо для работы deep link авторизации

**Примечание**: Приложение использует PKCE (Proof Key for Code Exchange) flow без client secret, что соответствует требованиям безопасности для desktop приложений.

## Тестирование

Проект использует многоуровневую стратегию тестирования с разными типами тестов для разных целей.

### Типы тестов и стратегия мокирования

#### 1. Модульные тесты (Unit Tests)
- **Расположение**: `tests/unit/`
- **Моки**: ✅ Все внешние зависимости мокируются (Electron, fs, БД, сеть)
- **Цель**: Изолированное тестирование логики отдельных функций/классов
- **Скорость**: < 100ms на тест

#### 2. Property-Based тесты
- **Расположение**: `tests/property/`
- **Моки**: ✅ Все внешние зависимости мокируются (аналогично модульным)
- **Цель**: Проверка инвариантов на множестве входных данных (100+ итераций)
- **Библиотека**: fast-check

#### 3. Интеграционные тесты
- **Расположение**: `tests/integration/`
- **Моки**: ❌ **НЕ мокируется Electron** - используется реальный Electron
- **Цель**: Проверка взаимодействия компонентов с реальным Electron
- **Особенность**: Может показывать окна на экране

#### 4. Функциональные тесты
- **Расположение**: `tests/functional/`
- **Моки**: ❌ **НЕ мокируется Electron** - используется реальный Electron
- **Цель**: End-to-end проверка сценариев с реальным Electron
- **Особенность**: Показывает реальные окна на экране

### Быстрый запуск тестов

```bash
# Валидация проекта (модульные + property-based тесты)

# Запуск всех тестов (включая интеграционные и функциональные)
```

### Интеграционные и функциональные тесты

⚠️ **ВАЖНО**: Интеграционные и функциональные тесты используют реальный Electron и могут показывать окна на экране!

```bash
# Запуск интеграционных тестов (реальный Electron)
npm run test:integration

# Запуск функциональных тестов (реальный Electron, показывает окна)

```

**Когда запускать интеграционные и функциональные тесты:**
- ✅ При явной необходимости проверить реальное поведение Electron
- ✅ Перед релизом новой версии
- ✅ При отладке проблем с окнами или IPC
- ❌ НЕ запускать автоматически в фоновом режиме
- ❌ НЕ запускать во время активной работы
- ❌ НЕ включены в `npm run validate`

**Требования для запуска:**
- Графическая среда (X11/Wayland на Linux, Window Server на macOS)
- Electron установлен как зависимость
- На Linux может потребоваться `xvfb` для headless режима

## Авторизация

Приложение использует Google OAuth 2.0 для авторизации пользователей:

### Процесс авторизации

1. **Первый запуск**: При первом запуске приложения отображается экран входа
2. **Клик "Continue with Google"**: Открывается системный браузер с формой авторизации Google
3. **Авторизация в браузере**: Пользователь входит в свой Google аккаунт и предоставляет разрешения
4. **Deep Link Callback**: Google перенаправляет на `clerkly://oauth/callback` с authorization code
5. **Обмен кода на токены**: Приложение обменивает код на access и refresh токены
6. **Сохранение токенов**: Токены безопасно сохраняются в локальной SQLite базе данных
7. **Главный экран**: Открывается главное окно приложения

### Безопасность авторизации

- **PKCE Flow**: Используется PKCE (Proof Key for Code Exchange) без client secret
- **Локальное хранение**: Токены хранятся только локально в зашифрованной SQLite базе
- **Автоматическое обновление**: Access токены автоматически обновляются при истечении
- **Deep Link**: Используется custom protocol `clerkly://` вместо localhost сервера

### API авторизации

Renderer process имеет доступ к следующим методам через безопасный IPC мост:

```typescript
// Начать процесс авторизации
await window.api.auth.startLogin();

// Получить текущий статус авторизации
const status = await window.api.auth.getAuthStatus();
// Возвращает: { authorized: boolean, error?: string }

// Выйти из системы
await window.api.auth.logout();
```

### Обработка ошибок

Приложение обрабатывает следующие типы ошибок авторизации:

- **popup_closed_by_user**: Пользователь закрыл окно браузера до завершения авторизации
- **access_denied**: Пользователь отказал в доступе к Google аккаунту
- **network_error**: Проблемы с сетевым подключением
- **invalid_grant**: Сессия авторизации истекла
- **csrf_attack_detected**: Обнаружена попытка CSRF атаки

Для каждой ошибки отображается понятное сообщение с рекомендациями по устранению.

## Архитектура

Приложение следует стандартной архитектуре Electron с разделением на процессы:

### Main Process (Основной процесс)
Управляет жизненным циклом приложения, окнами и бизнес-логикой:
- **Window Manager** - Создание и управление окнами с нативным Mac OS X интерфейсом
- **Lifecycle Manager** - Управление запуском, активацией и завершением приложения
- **Data Manager** - Локальное хранение данных с использованием SQLite
- **Migration Runner** - Управление миграциями схемы базы данных
- **IPC Handlers** - Обработка IPC запросов от Renderer Process

### Renderer Process (Процесс отрисовки)
Отвечает за UI и взаимодействие с пользователем:
- **React Application** - Современный UI с компонентным подходом
- **Top Navigation** - Навигация между разделами приложения
- **AI Agent Panel** - Панель взаимодействия с AI ассистентом
- **Dashboard** - Главный экран с обзором задач и встреч
- **Calendar View** - Календарь встреч и событий
- **Tasks View** - Управление задачами и проектами
- **Contacts** - Управление контактами
- **Settings** - Настройки приложения

### Preload Script (Предзагрузочный скрипт)
Обеспечивает безопасную IPC коммуникацию между процессами через `contextBridge`:
- Изолирует Renderer Process от прямого доступа к Node.js API
- Предоставляет безопасный API для взаимодействия с Main Process

### Хранение данных
- **SQLite** база данных для локального хранения пользовательских данных
- **Система миграций** для управления изменениями схемы базы данных
- Все данные хранятся локально на машине пользователя

## Тестирование

Проект использует комплексный подход к тестированию с четырьмя типами тестов:

### Типы тестов

1. **Модульные тесты (Unit Tests)**
   - Проверяют конкретные примеры и граничные случаи
   - Тестируют отдельные компоненты в изоляции
   - **Мокируют Electron API** для быстрого выполнения
   - Расположение: `tests/unit/`
   - Скорость: < 100ms на тест

2. **Property-based тесты**
   - Проверяют универсальные свойства на множестве входных данных
   - Автоматически генерируют тестовые данные (минимум 100 итераций)
   - **Мокируют Electron API** для быстрого выполнения
   - Используют библиотеку `fast-check`
   - Расположение: `tests/property/`

3. **Интеграционные тесты**
   - Проверяют взаимодействие компонентов
   - **Используют реальный Electron** (не мокируют)
   - Могут показывать окна на экране
   - Расположение: `tests/integration/`

4. **Функциональные тесты**
   - Проверяют end-to-end сценарии
   - **Используют реальный Electron** (не мокируют)
   - ⚠️ Показывают окна на экране - запускать только вручную!
   - Расположение: `tests/functional/`

### Требования к покрытию

- **Минимум 85%** покрытие строк кода
- **100%** покрытие для критических компонентов:
  - Data Manager
  - Lifecycle Manager
  - IPC Handlers
  - Window State Manager

### Структура тестов

Все тесты следуют единому формату с структурированными комментариями:

```typescript
   Assertions: описание ожидаемых результатов
   Requirements: testing.1.1, testing.2.3 */

  // Тест реализация
```

   ```bash
   npm run test:watch   # Автоматический запуск тестов при изменениях
   ```

3. **Перед коммитом**:
   ```bash
   npm run lint:fix     # Автоматическое исправление проблем линтинга
   npm run format       # Форматирование кода
   npm run validate     # Полная валидация
   ```

4. **Перед релизом**:
   ```bash
   npm run validate            # Полная валидация
   npm run test:functional     # Функциональные тесты (вручную)
   npm run package             # Создание дистрибутива
   ```

### Отладка

- **Main Process**: Используйте `console.log()` - вывод будет в терминале
- **Renderer Process**: Используйте DevTools (View → Toggle Developer Tools в меню приложения)
- **Preload Script**: Логи доступны в DevTools консоли

### Известные проблемы

- При первой установке `better-sqlite3` может потребоваться компиляция нативных модулей
- Функциональные тесты могут конфликтовать с другими открытыми окнами Electron
- На старых версиях Mac OS X (10.13-10.14) возможны проблемы с vibrancy эффектами

## Производительность

Приложение оптимизировано для следующих показателей:

- **Время запуска**: < 3 секунды на современных Mac системах
- **Отклик UI**: < 100ms для всех операций интерфейса
- **Операции с данными**: < 50ms для простых операций чтения/записи
- **Индикаторы загрузки**: Автоматически показываются для операций > 200ms

## Безопасность

- **Локальное хранение**: Все данные хранятся локально на машине пользователя
- **Изоляция процессов**: Renderer Process изолирован от Node.js API через contextBridge
- **Валидация IPC**: Все IPC запросы валидируются перед обработкой
- **Таймауты**: IPC операции имеют таймаут 10 секунд для предотвращения зависания

## Устранение неполадок

### Проблемы при установке

**Ошибка компиляции better-sqlite3**:
```bash
# Переустановка с пересборкой нативных модулей
npm rebuild better-sqlite3
```

**Ошибки прав доступа**:
```bash
# Очистка кэша npm и переустановка
npm cache clean --force
rm -rf node_modules package-lock.json
npm install
```

### Проблемы при запуске

**Приложение не запускается**:
1. Проверьте версию Node.js: `node --version` (должна быть >= 18.0.0)
2. Пересоберите проект: `npm run build`
3. Проверьте логи в терминале на наличие ошибок

**База данных повреждена**:
- Приложение автоматически создаст backup и новую базу данных
- Backup сохраняется в директории данных приложения

### Проблемы с тестами

**Тесты не проходят**:
```bash
# Очистка кэша Jest
npm test -- --clearCache

# Запуск тестов с подробным выводом
npm test -- --verbose
```

**Функциональные тесты зависают**:
- Закройте все другие окна Electron
- Убедитесь, что нет других запущенных экземпляров приложения
