# Руководство для Агентов

## Быстрый Справочник Команд

### Валидация
- `npm run validate` - полная валидация (TypeScript, ESLint, Prettier, модульные, property-based тесты)
- `npm run validate:verbose` - валидация с подробным выводом

### Тесты
- `npm test` - модульные + property-based тесты
- `npm run test:unit` - только модульные тесты
- `npm run test:property` - только property-based тесты
- `npm run test:functional` - функциональные тесты (показывают окна!)

### Отладка
- `npm run test:unit -- path/to/test.ts -t "test name"` - запустить конкретный тест
- `npm run test:functional:debug -- test.spec.ts` - функциональные с остановкой на первой ошибке
- `npx playwright show-report` - посмотреть отчет функциональных тестов

## Обязательный Workflow Работы

**КРИТИЧЕСКИ ВАЖНО**: Каждая задача ДОЛЖНА выполняться по следующему workflow:

### 1. Сбор Контекста

**Что делать:**
- Прочитать требования (`.kiro/specs/*/requirements.md`)
- Прочитать дизайн (`.kiro/specs/*/design.md`)
- Прочитать список задач (`.kiro/specs/*/tasks.md`)
- Изучить существующий код, связанный с задачей
- Изучить существующие тесты (модульные, property-based, функциональные)

**Цель:** Понять контекст задачи, существующую архитектуру и требования.

### 2. Формирование Плана

**Что делать:**
- Сформировать детальный план действий
- Определить, какие файлы нужно изменить/создать
- Определить, какие тесты нужно написать/обновить
- Оценить риски и зависимости
- **СОГЛАСОВАТЬ ПЛАН С ПОЛЬЗОВАТЕЛЕМ** перед началом работы

**Цель:** Убедиться, что подход правильный и согласован с пользователем.

**Формат плана:**
```
План действий:
1. Изменить файл X - добавить функцию Y
2. Создать тест Z для проверки функции Y
3. Обновить документацию в requirements.md
4. Запустить валидацию

Ожидаемый результат: [описание]
Риски: [если есть]
```

### 3. Выполнение Задачи

**Что делать:**
- Выполнять план пошагово
- После каждого значимого изменения:
  - Запускать модульные тесты (`npm run test:unit`)
  - Запускать property-based тесты (`npm run test:property`)
  - Проверять, что изменения не сломали существующую функциональность
- Писать код с комментариями требований
- Писать тесты с правильной структурой (Preconditions, Action, Assertions, Requirements)

**Цель:** Постепенно реализовать задачу с валидацией на каждом этапе.

### 4. Завершение Задачи

**Что делать:**
- Запустить полную валидацию (`npm run validate`)
- Убедиться, что все проверки проходят:
  - ✅ TypeScript компиляция
  - ✅ ESLint
  - ✅ Prettier
  - ✅ Модульные тесты
  - ✅ Property-based тесты
  - ✅ Покрытие кода
- **ЗАПРОСИТЬ У ПОЛЬЗОВАТЕЛЯ** запуск функциональных тестов:
  > "Задача выполнена. Запустить функциональные тесты? (они покажут окна на экране)"

**Цель:** Убедиться, что задача выполнена полностью и корректно.

### 5. Отчет об Успехе

**Что делать:**
- Дать структурированную сводку о выполненной работе
- Указать, что было изменено (файлы, тесты, документация)
- Указать, что осталось сделать (если есть)
- НЕ создавать отдельные файлы с отчетами

**Формат отчета:**
```
Задача выполнена.

Реализовано:
- [пункт 1]
- [пункт 2]

Изменены файлы:
- [файл 1]
- [файл 2]

Добавлены/обновлены тесты:
- [тест 1]
- [тест 2]

Осталось сделать:
- [пункт 1] (если есть)
- [пункт 2] (если есть)
```

**Цель:** Информировать пользователя о результатах работы.

## Определения и Термины

- **Максимальное покрытие кода**: Минимум 85% покрытия строк кода тестами, стремление к 90%+
- **Краевые случаи**: Тестирование граничных значений, пустых данных, null/undefined, максимальных/минимальных значений
- **Исключительные ситуации**: Тестирование ошибок сети, недоступности ресурсов, таймаутов, некорректных входных данных
- **Компонент**: Логическая единица системы (класс, модуль, функция, UI-компонент)

## Стратегия Тестирования

### Типы Тестов

#### Модульные Тесты (Unit Tests)
- **Расположение**: `tests/unit/**/*.test.ts`
- **Моки**: ✅ ВСЕ внешние зависимости мокируются (Electron, fs, БД, сеть)
- **Цель**: Изолированное тестирование логики отдельных функций/классов
- **Скорость**: < 100ms на тест

#### Property-Based Тесты
- **Расположение**: `tests/property/**/*.property.test.ts`
- **Моки**: ✅ ВСЕ внешние зависимости мокируются (аналогично модульным)
- **Цель**: Проверка инвариантов на множестве входных данных
- **Итерации**: Минимум 100 на тест

#### Функциональные Тесты
- **Расположение**: `tests/functional/**/*.spec.ts`
- **Фреймворк**: Playwright для Electron
- **Моки**: ❌ **НЕ мокировать Electron API** - используется реальный Electron
- **Реальные компоненты**: Electron, внутренние классы, БД, файловая система
- **Особенность**: Показывают реальные окна на экране
- **Цель**: End-to-end проверка пользовательских сценариев

**ВАЖНО**: Функциональные тесты покрывают пользовательские сценарии (User Stories) из requirements.md!

### Подготовка к Запуску Тестов

**КРИТИЧЕСКИ ВАЖНО**: Перед запуском тестов НЕОБХОДИМО перебилдить нативные модули для текущей версии Node.js.

```bash
# Перебилдить better-sqlite3 для текущей версии Node.js
npm run rebuild:node
```

**Когда нужно запускать rebuild:**
- После переключения версии Node.js
- После установки/обновления зависимостей (`npm install`)
- При ошибках типа `ERR_DLOPEN_FAILED` или `MODULE_NOT_FOUND`
- Перед первым запуском тестов после клонирования репозитория

**Автоматический rebuild:**
Команда `npm test` автоматически выполняет `npm run rebuild:node`.

Для отдельных типов тестов rebuild нужно запускать вручную:
```bash
npm run rebuild:node && npm run test:unit
npm run rebuild:node && npm run test:property
npm run rebuild:node && npm run test:functional
```

### Правила Запуска Функциональных Тестов

**КРИТИЧЕСКИ ВАЖНО**: Функциональные тесты используют реальный Electron и показывают окна на экране.

**Когда НЕ запускать:**
- ❌ При автоматической валидации (`npm run validate`)
- ❌ При обычной разработке без явного запроса пользователя
- ❌ В фоновом режиме во время работы пользователя

**Сценарии запуска:**

1. **"Запусти все тесты"** - пользователь явно просит запустить ВСЕ тесты
   - ✅ Запускать функциональные тесты БЕЗ дополнительного подтверждения
   - ⚠️ Предупредить ПЕРЕД запуском: "Запускаю все тесты, включая функциональные (покажут окна на экране)"
   - 📋 Порядок: `npm run validate` → `npm run test:functional`

2. **"Запусти функциональные тесты"** - пользователь явно просит функциональные тесты
   - ✅ Запускать БЕЗ дополнительного подтверждения
   - ⚠️ Предупредить ПЕРЕД запуском: "Функциональные тесты покажут окна на экране"

3. **Завершение задачи** - агент завершил работу и запустил валидацию
   - ❌ НЕ запускать функциональные тесты автоматически
   - ✅ СПРОСИТЬ: "Задача выполнена. Запустить функциональные тесты? (они покажут окна на экране)"

### Алгоритм Запуска Функциональных Тестов в Фоне

**КРИТИЧЕСКИ ВАЖНО**: Функциональные тесты ДОЛГИЕ (90+ тестов, ~2-3 минуты). НЕ использовать `executeBash` с timeout - это создаст второй экземпляр процесса!

#### Шаг 1: Запуск в Фоне

```bash
# ПРАВИЛЬНО: Использовать controlBashProcess для фонового запуска
controlBashProcess(action="start", command="npm run test:functional")
```

**ЗАПРЕЩЕНО:**
- ❌ `executeBash` с `timeout` - создаст второй экземпляр при таймауте
- ❌ Запускать несколько раз подряд - проверяй статус сначала!

#### Шаг 2: Проверка Статуса

```bash
# Проверить список фоновых процессов
listProcesses()
```

**Ожидаемый вывод:**
```
processes: [
  {
    processId: 123,
    command: "npm run test:functional",
    path: "/path/to/project",
    status: "running"
  }
]
```

**Если процесс НЕ найден:**
- Значит тесты УЖЕ завершились
- Переходи к Шагу 3 для просмотра результатов

**Если процесс найден со статусом "running":**
- Тесты ЕЩЕ выполняются
- НЕ запускай их снова!
- Подожди или сообщи пользователю

#### Шаг 3: Просмотр Результатов

**Вариант A: Процесс еще выполняется**
```bash
# Посмотреть последние 50 строк вывода
getProcessOutput(processId=123, lines=50)
```

**Вариант B: Процесс завершился**
```bash
# Посмотреть последние строки вывода (результаты тестов)
# Используй обычный executeBash для чтения файла отчета
cat playwright-report/index.html  # HTML отчет
# ИЛИ
npx playwright show-report  # Открыть отчет в браузере
```

**Поиск результатов в выводе:**
```bash
# Искать строки с результатами тестов
grep -E "(Test Suites|Tests:|passed|failed)" output.txt
```

#### Шаг 4: Остановка Процесса (если нужно)

```bash
# Остановить фоновый процесс
controlBashProcess(action="stop", processId=123)
```

**Когда останавливать:**
- Пользователь попросил остановить
- Нужно запустить тесты заново с другими параметрами
- Процесс завис (статус "unknown")

### Полный Пример Workflow

```python
# 1. Запустить тесты в фоне
result = controlBashProcess(action="start", command="npm run test:functional")
processId = result.processId

# Сообщить пользователю
print(f"Функциональные тесты запущены в фоне (processId={processId})")
print("Это займет ~2-3 минуты. Тесты покажут окна на экране.")

# 2. Через некоторое время проверить статус
processes = listProcesses()
running = [p for p in processes if p.processId == processId and p.status == "running"]

if running:
    print("Тесты еще выполняются...")
    # Показать последние строки
    output = getProcessOutput(processId=processId, lines=20)
    print(output)
else:
    print("Тесты завершились. Смотрю результаты...")
    # Посмотреть отчет
    executeBash("npx playwright show-report")
```

### Частые Ошибки

**Ошибка 1: "Command timed out after 120000ms"**
- **Причина:** Использовал `executeBash` с timeout вместо `controlBashProcess`
- **Решение:** Использовать `controlBashProcess(action="start")` для фоновых процессов

**Ошибка 2: Запустил тесты дважды**
- **Причина:** Не проверил `listProcesses()` перед повторным запуском
- **Решение:** ВСЕГДА проверяй `listProcesses()` перед запуском

**Ошибка 3: Не могу найти результаты**
- **Причина:** Процесс завершился, но не знаю где результаты
- **Решение:** Результаты в `playwright-report/index.html` или используй `npx playwright show-report`

### Порядок Запуска Тестов

**При команде "запусти все тесты":**
1. Сначала валидация: `npm run validate` (включает TypeScript, ESLint, Prettier, модульные и property-based тесты)
2. Затем функциональные тесты: `npm run test:functional`

**Причина:** Если упадут быстрые модульные тесты или проверки кода, нет смысла запускать медленные функциональные тесты.

**При падении тестов:**
- ❌ НЕ продолжать запуск следующих типов тестов
- ✅ Остановиться и сообщить пользователю о проблеме
- ✅ Предложить запустить только упавшие тесты для отладки

### Стратегия Отладки Упавших Тестов

#### Модульные Тесты

**Шаг 1:** Запустить ТОЛЬКО упавший тест
```bash
npm run test:unit -- tests/unit/auth/UserProfileManager.test.ts -t "specific test name"
```

**Шаг 2:** Если тест все еще падает, запустить с подробным выводом
```bash
npm run test:unit -- tests/unit/auth/UserProfileManager.test.ts -t "specific test name" --verbose
```

**Шаг 3:** Проанализировать вывод и исправить проблему

#### Property-Based Тесты

**Шаг 1:** Запустить упавший property-based тест
```bash
npm run test:property -- tests/property/auth/OAuthConfig.property.test.ts -t "specific test name"
```

**Шаг 2:** Если тест падает на конкретном примере, скопировать этот пример из вывода

**Шаг 3:** Создать модульный тест с этим конкретным примером для отладки

**Шаг 4:** Исправить код

**Шаг 5:** Убедиться, что property-based тест теперь проходит

**ЗАПРЕЩЕНО:**
- ❌ Запускать все тесты заново для отладки одного упавшего теста
- ❌ Отключать тест без попытки исправления

### Параллельный Запуск Тестов

**РАЗРЕШЕНО:**
- ✅ Параллельный запуск модульных тестов внутри `npm run test:unit` (Jest делает это автоматически)
- ✅ Параллельный запуск property-based тестов внутри `npm run test:property`

**ЗАПРЕЩЕНО:**
- ❌ Запускать `npm run test:unit` и `npm run test:property` одновременно в разных терминалах
- ❌ Запускать функциональные тесты параллельно с любыми другими тестами
- ❌ Запускать несколько экземпляров `npm run test:functional` одновременно

**ПРИЧИНА:** Функциональные тесты используют реальную БД и файловую систему, что может конфликтовать с другими тестами.

### Правила Покрытия Кода

**Минимальные требования:**
- Строки (Statements): 85%
- Ветви (Branches): 80%
- Функции (Functions): 85%
- Строки кода (Lines): 85%

**Исключения из покрытия:**
- Файлы миграций БД (`migrations/**`)
- Конфигурационные файлы
- Типы и интерфейсы без логики

**При недостаточном покрытии:**
1. Определить непокрытые строки: `npm run test:coverage`
2. Добавить модульные тесты для непокрытых функций
3. Добавить property-based тесты для сложной логики
4. Запустить валидацию повторно

## Правила Документации

### Язык Спецификаций
- **Требования (requirements.md)**: Русский язык
- **Дизайн (design.md)**: Русский язык
- **Списки задач (tasks.md)**: Русский язык
- **Комментарии в коде**: Английский язык
- **Названия файлов и переменных**: Английский язык

### Именование Компонентов
- **Названия компонентов** должны быть на английском языке
- **Не используйте подчеркивания** в названиях компонентов
- **При ссылке на компоненты** всегда заключайте их в кавычки

**Примеры:**
- ✅ "Main Process", "OAuth Flow", "Sidebar State"
- ❌ Главный_Процесс, OAuth_Flow, Sidebar_State

### Покрытие Пользовательских Сценариев

**КРИТИЧЕСКИ ВАЖНО**: Каждый пользовательский сценарий (User Story) ДОЛЖЕН быть покрыт функциональным тестом.

**Формат секции "Функциональные Тесты":**
```markdown
#### Функциональные Тесты

- `tests/functional/app-lifecycle.spec.ts` - "should launch application successfully"
- `tests/functional/auth-flow.spec.ts` - "should show login screen on first launch"
```

**Требования:**
- Каждый пользовательский сценарий в requirements.md ДОЛЖЕН иметь минимум один функциональный тест
- Каждое требование с пользовательским сценарием ДОЛЖНО содержать секцию "Функциональные Тесты"
- Указывать путь к файлу теста и название теста
- Функциональные тесты используют Playwright и НЕ мокируют Electron API

### Таблица Покрытия Требований

**КРИТИЧЕСКИ ВАЖНО**: Каждый документ дизайна (design.md) ДОЛЖЕН содержать актуальную таблицу покрытия требований тестами.

**Формат:**
```markdown
### Покрытие Требований

| Требование | Модульные Тесты | Property-Based Тесты | Функциональные Тесты |
|------------|-----------------|----------------------|----------------------|
| feature.1.1 | ✓ | ✓ | - |
| feature.1.2 | ✓ | - | ✓ |
| feature.2.1 | ✓ | ✓ | ✓ |
```

**Требования:**
- Таблица ДОЛЖНА присутствовать в каждом документе дизайна
- Таблица ДОЛЖНА включать ВСЕ требования из requirements.md
- Использовать `✓` для покрытия, `-` для отсутствия покрытия

## Правила Написания Кода

### Логирование и Обработка Ошибок

**КРИТИЧЕСКИ ВАЖНО**: В проекте используется централизованная система логирования и обработки ошибок.

#### Logger

**Спецификация:** `.kiro/specs/clerkly/requirements.md` (clerkly.3)

**Правила использования:**
- Каждый класс создает свой экземпляр Logger с именем класса: `new Logger('ClassName')`
- Logger автоматически добавляет имя класса к сообщениям
- НЕ добавлять `[ClassName]` в сообщения вручную - это дублирование

**Пример:**
```typescript
// ✅ ПРАВИЛЬНО
this.logger = new Logger('UserProfileManager');
this.logger.info('User ID set: abc123');
// Вывод: [UserProfileManager] User ID set: abc123

// ❌ НЕПРАВИЛЬНО
this.logger.info('[UserProfileManager] User ID set: abc123');
// Вывод: [UserProfileManager] [UserProfileManager] User ID set: abc123
```

#### ErrorHandler

**Спецификация:** `.kiro/specs/error-notifications/requirements.md` (error-notifications.1)
**Дизайн:** `.kiro/specs/error-notifications/design.md`

**Правила использования:**
- Для ошибок в фоновых процессах использовать `ErrorHandler.handleBackgroundError(error, context)`
- ErrorHandler автоматически логирует ошибку и отправляет уведомление пользователю
- ErrorHandler автоматически фильтрует race condition ошибки (не показывает пользователю)

**Пример:**
```typescript
// ✅ ПРАВИЛЬНО - использовать ErrorHandler
async fetchProfile(): Promise<UserProfile | null> {
  try {
    // ... код ...
  } catch (error) {
    ErrorHandler.handleBackgroundError(error, 'Profile Loading');
    return null;
  }
}

// ❌ НЕПРАВИЛЬНО - локальное логирование ошибок
async fetchProfile(): Promise<UserProfile | null> {
  try {
    // ... код ...
  } catch (error) {
    this.logger.error('Failed to fetch profile:', error);
    return null;
  }
}
```

**Фильтрация ошибок (error-notifications.1.5):**
- "No user logged in" во время logout - НЕ показывается пользователю
- Отмененные операции (cancelled, aborted) - НЕ показываются пользователю
- Race condition ошибки - НЕ показываются пользователю
- Все ошибки ВСЕГДА логируются для отладки

### Комментарии с Требованиями

**КРИТИЧЕСКИ ВАЖНО**: Каждый фрагмент кода ДОЛЖЕН содержать комментарии со ссылками на требования.

**Формат:**
```typescript
// Requirements: requirement-id.1.1, requirement-id.1.2
function implementFeature() {
  // Код реализации
}
```

**Требования:**
- Каждая функция, класс, метод ДОЛЖНЫ иметь комментарий с требованиями
- Указывать только те требования, которые непосредственно реализует данный код
- Обновлять комментарии при изменении кода или требований

### Структура Тестов

**КРИТИЧЕСКИ ВАЖНО**: Каждый тест ДОЛЖЕН содержать структурированный комментарий.

**Формат:**
```typescript
/* Preconditions: описание начального состояния системы
   Action: описание выполняемого действия
   Assertions: описание ожидаемых результатов и проверок
   Requirements: requirement-id.1.1, requirement-id.1.2 */
it("should perform expected behavior", () => {
  // Тест реализация
});
```

**Требования:**
- **Preconditions**: Четко описать начальное состояние системы, моков, данных
- **Action**: Конкретно указать, какое действие выполняется в тесте
- **Assertions**: Детально описать все ожидаемые результаты и проверки
- **Requirements**: Перечислить все требования, которые покрывает данный тест
- Каждый тест ДОЛЖЕН содержать все четыре компонента

## Критические Правила

### Отключение Тестов

**КРИТИЧЕСКИ ВАЖНО**: Тесты НЕЛЬЗЯ отключать (skip) без явного подтверждения пользователя.

**АБСОЛЮТНЫЙ ЗАПРЕТ:**
- ❌ Агент НЕ ИМЕЕТ ПРАВА отключать тесты (использовать `.skip()`, `.only()`, комментировать тесты) без явного разрешения пользователя
- ❌ Агент НЕ ИМЕЕТ ПРАВА предлагать отключение тестов как решение проблемы
- ❌ Агент НЕ ИМЕЕТ ПРАВА отключать тесты "временно" или "для рефакторинга"

**Требования:**
- Перед отключением любого теста НЕОБХОДИМО:
  1. Объяснить пользователю, почему тест не может быть исправлен
  2. Предложить альтернативные решения
  3. Получить явное подтверждение от пользователя ("да, отключи тест")
- Всегда пытаться исправить тест, а не отключить его
- Если тест падает - это проблема кода или теста, которую нужно решить

**Альтернативы отключению:**
- Перенести в функциональные тесты (если требуется реальный Electron)
- Упростить тест, уменьшив количество итераций
- Улучшить изоляцию состояния между итерациями
- Исправить код, чтобы тест проходил
- Обновить тест, чтобы он соответствовал новому поведению

### Создание Временных Отчетов

**КРИТИЧЕСКИ ВАЖНО**: Агент НЕ ДОЛЖЕН создавать временные отчеты, summary файлы или документацию без явного запроса пользователя.

**Запрещенные действия:**
- ❌ Создание файлов типа `VALIDATION_REPORT.md`, `SUMMARY.md`, `WORK_REPORT.md` без запроса
- ❌ Создание markdown файлов для документирования выполненной работы
- ❌ Любые файлы, которые дублируют информацию из существующих спецификаций

**Правильный подход:**
- ✅ Предоставлять краткую устную сводку в конце работы (2-3 предложения)
- ✅ Обновлять существующие файлы спецификаций (tasks.md, design.md)
- ✅ Создавать отчеты ТОЛЬКО если пользователь явно попросил

### Использование Переменных Окружения

**КРИТИЧЕСКИ ВАЖНО**: Агент НЕ ДОЛЖЕН создавать новые переменные окружения без явного согласования с пользователем.

**Требования:**
- Агент НЕ ДОЛЖЕН добавлять новые переменные окружения (`process.env.VARIABLE_NAME`) без явного разрешения
- Перед добавлением переменной окружения НЕОБХОДИМО:
  1. Объяснить пользователю, зачем нужна переменная окружения
  2. Предложить альтернативные решения (аргументы командной строки, конфигурационные файлы)
  3. Получить явное подтверждение от пользователя
- Всегда пытаться использовать альтернативные подходы вместо переменных окружения

**Исключения:**
- Переменная уже существует в коде
- Пользователь явно попросил использовать переменную окружения
- Это стандартная переменная окружения (например, `NODE_ENV`, `PATH`)

## Валидация Изменений

**КРИТИЧЕСКИ ВАЖНО**: Каждое изменение в коде, спецификациях или тестах ДОЛЖНО быть провалидировано перед завершением работы.

### Валидация Спецификаций

Все спецификации ДОЛЖНЫ быть:
- **Полными**: Покрывать все аспекты функциональности
- **Непротиворечивыми**: Не содержать конфликтующих требований
- **Не избыточными**: Не дублировать требования между спецификациями

```bash
npm run validate
```

Скрипт автоматически выполняет все необходимые проверки в правильном порядке и исправляет проблемы ESLint/Prettier где возможно.

**Требования:**
- Все проверки ДОЛЖНЫ проходить без ошибок
- При падении проверок: исправить код или обновить тесты
- Новая функциональность ДОЛЖНА быть покрыта тестами
- **МАКСИМАЛЬНОЕ ПОКРЫТИЕ**: Код и требования ДОЛЖНЫ быть максимально покрыты тестами
- **КРАЕВЫЕ СЛУЧАИ**: Особое внимание ДОЛЖНО уделяться тестированию краевых случаев и исключительных ситуаций

Работа считается завершенной ТОЛЬКО когда:
- ✅ Все спецификации корректны и согласованы
- ✅ TypeScript компилируется без ошибок
- ✅ ESLint проходит без замечаний
- ✅ Prettier форматирование корректно
- ✅ Все модульные тесты проходят
- ✅ Покрытие тестами соответствует требованиям
- ✅ Все требования покрыты тестами
- ✅ Код содержит комментарии с требованиями

**Примечание**: Функциональные тесты НЕ являются обязательными для завершения работы. Они запускаются ТОЛЬКО при явной просьбе пользователя.

## Приоритеты при Конфликтах

Если правила противоречат друг другу, следовать приоритетам:

## Частые Ошибки и Решения

### Ошибка: "Cannot find module 'better-sqlite3'"
**Решение:** `npm run rebuild:node`

**Причина:** Нативный модуль не собран для текущей версии Node.js

### Ошибка: Тесты падают с таймаутом
**Решение:** Увеличить таймаут: `npm run test:unit -- --testTimeout=10000`

**Причина:** Тест выполняется дольше стандартного таймаута (5000ms)

### Ошибка: Property-based тест падает нестабильно
**Решение:** 
1. Проверить изоляцию состояния между итерациями
2. Убедиться, что моки сбрасываются после каждой итерации (`jest.clearAllMocks()`)
3. Уменьшить количество итераций для отладки
4. Добавить seed для воспроизводимости: `fc.assert(fc.property(...), { seed: 42 })`

**Причина:** Состояние между итерациями не изолировано или есть race condition

### Ошибка: Функциональные тесты не запускаются
**Решение:** 
1. `npm run rebuild:electron`
2. `npm run build`
3. Проверить, что Electron установлен: `npx electron --version`

**Причина:** Нативные модули не собраны для Electron или приложение не собрано

### Ошибка: ESLint или Prettier падают
**Решение:** 
1. `npm run lint:fix` - автоматическое исправление ESLint
2. `npm run format` - автоматическое форматирование Prettier
3. `npm run validate` - запустить полную валидацию с автоисправлением

**Причина:** Код не соответствует стандартам форматирования

### Ошибка: Покрытие кода ниже требуемого
**Решение:**
1. `npm run test:coverage` - посмотреть отчет о покрытии
2. Открыть `coverage/lcov-report/index.html` в браузере
3. Найти непокрытые строки
4. Добавить тесты для непокрытых функций
5. Запустить `npm run validate` повторно

**Причина:** Недостаточно тестов для нового кода
