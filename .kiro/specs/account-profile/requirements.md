# Документ Требований: Account Profile

## Введение

Данный документ описывает требования к отображению и управлению профилем пользователя в приложении Clerkly, включая загрузку данных из Google OAuth и синхронизацию с локальной базой данных.

## Глоссарий

- **Account Block** - Блок интерфейса для отображения информации о профиле пользователя
- **User Profile** - Профиль пользователя, содержащий данные из Google OAuth (имя, email, аватар)
- **Google UserInfo API** - API Google для получения информации о пользователе
- **Read-only Fields** - Поля, которые пользователь не может редактировать в приложении
- **Synchronous Loading** - Синхронная загрузка профиля во время авторизации (блокирующая операция)
- **Loader** - Индикатор загрузки, отображаемый во время асинхронных операций
- **LoginError Component** - Компонент для отображения ошибок авторизации
- **Access Token** - Токен доступа OAuth, используемый для авторизации API запросов
- **Refresh Token** - Токен обновления OAuth, используемый для получения нового access token

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `.kiro/specs/clerkly/requirements.md`).

**Применение к Account Profile:**
- Данные профиля пользователя сохраняются в базе данных после загрузки из Google UserInfo API
- UI отображает профиль из базы данных
- Профиль автоматически обновляется при изменении данных в Google
- Данные профиля сохраняются даже после logout (очищаются только токены)

## Требования

### 1. Блок Account (Профиль Пользователя)

**ID:** account-profile.1

**User Story:** Как пользователь, я хочу видеть информацию о моем Google аккаунте в приложении.

**Зависимости:** google-oauth-auth.3, google-oauth-auth.5

#### Критерии Приемки

1.1. "Account Block" ДОЛЖЕН отображать следующие поля из профиля пользователя:
   - Имя пользователя (name)
   - Email адрес (email)

1.2. "Account Block" ДОЛЖЕН НЕ позволять пользователю редактировать поля профиля (read-only)

1.3. Система ДОЛЖНА загружать данные профиля из Google UserInfo API:
   - **Синхронно** во время успешной авторизации (после обмена authorization code на токены, во время отображения loader - см. google-oauth-auth.15)
   - При запуске приложения (если пользователь авторизован)
   - При обновлении access token (refresh)

1.4. КОГДА профиль загружается синхронно во время авторизации, ТО:
   - Система ДОЛЖНА показывать loader во время получения профиля
   - При успехе: показать главный интерфейс приложения (Dashboard)
   - При ошибке: очистить токены И показать LoginError компонент с errorCode 'profile_fetch_failed'

1.5. КОГДА профиль не может быть получен во время авторизации, ТО авторизация ДОЛЖНА считаться неуспешной (токены очищаются, пользователь видит LoginError)

**Примечание:** Отображение данных из базы и автоматическое обновление UI при изменении данных следуют из архитектурного принципа. Данные профиля пользователя сохраняются в базе данных даже после logout - очищаются только токены авторизации.

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Редактирование данных профиля пользователя в приложении
- Загрузка и изменение аватара пользователя
- Отображение дополнительных полей профиля (телефон, адрес и т.д.)
- Синхронизация изменений профиля обратно в Google
