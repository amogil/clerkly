# Документ Требований - Google OAuth Аутентификация

## Введение

Этот документ определяет требования для аутентификации Google OAuth 2.0 с поддержкой PKCE, включая безопасное управление токенами, поток авторизации и обработку пользовательских сессий.

## Глоссарий

- **OAuth Flow**: Процесс авторизации Google OAuth 2.0 с PKCE
- **Authorization Gate**: Экран авторизации, показываемый когда пользователь не аутентифицирован
- **PKCE**: Расширение безопасности Proof Key for Code Exchange
- **Authorization Completion Page**: Страница, показываемая после OAuth callback

## Зависимости

- **Требует**: platform-foundation (для IPC каналов), data-storage (для сохранения токенов)
- **Предоставляет**: Состояние аутентификации пользователя, безопасное управление токенами
- **IPC Каналы**: "auth:open-google", "auth:result"

## Требования

### Требование 1: Поток OAuth Авторизации

**Пользовательская История:** Как пользователь, я хочу безопасно аутентифицироваться с моим Google аккаунтом, чтобы получить доступ к приложению с существующими учетными данными.

#### Критерии Приемки

1. ЕСЛИ приложение не авторизовано с Google аккаунтом пользователя, "Application" ДОЛЖНО показать единственный экран авторизации и НЕ ДОЛЖНО показывать меню сайдбара
2. "Authorization Screen" ДОЛЖЕН включать кнопку "Войти через Google"
3. КОГДА нажимается кнопка "Войти через Google", "System Browser" ДОЛЖЕН открыться для входа в Google
4. ПОСЛЕ успешной авторизации, "Application" ДОЛЖНО вернуться к приложению и отобразить полный интерфейс
5. "Authorization Screen" ДОЛЖЕН оставаться видимым до тех пор, пока OAuth callback не укажет на успех
6. ЕСЛИ авторизация не удалась, "Application" ДОЛЖНО показать сообщение об ошибке и оставить кнопку входа в Google доступной
7. "OAuth Implementation" ДОЛЖНА использовать loopback redirect URI в форме "http://127.0.0.1:<port>" или "http://[::1]:<port>" для настольной авторизации
8. "Authorization Configuration" ДОЛЖНА быть определена в src/auth/auth_google.ts как основной источник истины
9. "Authorization Success" ДОЛЖЕН считаться успешным только когда OAuth callback включает параметр "code"

### Требование 2: Управление Токенами и Безопасность

**Пользовательская История:** Как пользователь, я хочу чтобы моя аутентификация была безопасной и постоянной, чтобы не приходилось часто повторно аутентифицироваться, сохраняя при этом безопасность.

#### Критерии Приемки

1. "Application" ДОЛЖНО сохранять состояние авторизации между запусками и восстанавливать его при старте
2. "OAuth Tokens" ДОЛЖНЫ храниться в SQLite в зашифрованном виде; хранение токенов в открытом тексте запрещено
3. "OAuth Access Tokens" ДОЛЖНЫ обновляться автоматически в фоновом режиме когда возможно
4. "User Interface" ДОЛЖЕН предоставлять явное действие выхода, которое очищает сохраненную сессию и токены
5. "Runtime Authorization Data" ДОЛЖНЫ храниться в базе данных SQLite; значения по умолчанию конфигурации могут браться из src/auth/auth_google.ts

### Требование 3: Реализация Безопасности PKCE

**Пользовательская История:** Как пользователь, заботящийся о безопасности, я хочу чтобы поток OAuth использовал современные практики безопасности, чтобы моя аутентификация была защищена от атак.

#### Критерии Приемки

1. "Google OAuth" ДОЛЖЕН использовать PKCE (code_verifier/code_challenge с S256) для настольной авторизации
2. "PKCE Code Verifier" ДОЛЖЕН храниться только в памяти для активной попытки авторизации и очищаться после обмена токенами или неудачи
3. "OAuth Authorization Requests" ДОЛЖНЫ включать случайное значение state и callback ДОЛЖЕН проверить, что state соответствует ожидающей попытке авторизации перед обменом кода
4. "Application" ДОЛЖНО включать OAuth client secret при обмене кодов авторизации и обновлении токенов с Google
5. "OAuth Client Secret" ДОЛЖЕН быть определен в src/auth/auth_google.ts и использоваться для обмена токенами и обновления

### Требование 4: Завершение Авторизации и Обработка Ошибок

**Пользовательская История:** Как пользователь, я хочу четкую обратную связь во время процесса авторизации, чтобы понимать что происходит и иметь возможность восстановиться после ошибок.

#### Критерии Приемки

1. ЕСЛИ пользователь закрывает вкладку браузера во время авторизации Google, "Application" ДОЛЖНО остаться в неавторизованном состоянии, оставить кнопку "Войти через Google" доступной и позволить начать поток авторизации снова
2. "Authorization Completion Page" ДОЛЖНА попытаться закрыть себя с setTimeout(() => window.close(), 300)
3. "Authorization Completion Page" ДОЛЖНА отрендерить логотип Clerkly и стилизованное сообщение об успехе, сохраняя поведение автозакрытия
4. "Authorization Completion Page" ДОЛЖНА упомянуть, что пользователь может вернуться к приложению Clerkly
5. КОГДА OAuth callback включает ошибку или отсутствует код авторизации, "Completion Page" ДОЛЖНА отрендерить сообщение о неудаче и НЕ ДОЛЖНА показывать текст об успехе
6. "OAuth Error Codes" ДОЛЖНЫ быть преобразованы в понятные пользователю сообщения и НЕ ДОЛЖНЫ отображать сырые значения типа access_denied

### Требование 5: Интеграция IPC

**Пользовательская История:** Как разработчик, я хочу безопасную коммуникацию между компонентами аутентификации, чтобы архитектура оставалась поддерживаемой и безопасной.

#### Критерии Приемки

1. "IPC Channels" "auth:open-google" и "auth:result" ДОЛЖНЫ быть частью стабильного контракта авторизации и НЕ ДОЛЖНЫ изменяться без обновления требований
2. "Authentication State" ДОЛЖНО передаваться другим процессам через безопасные IPC каналы
3. "Auth IPC Contract" ДОЛЖЕН быть документирован и версионирован для интеграции с другими фичами
