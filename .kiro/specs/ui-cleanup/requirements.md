# Документ Требований: Очистка Интерфейса

## Введение

Данная спецификация описывает полную переделку пользовательского интерфейса приложения с целью упрощения до минимального состояния. После реализации приложение будет показывать только экран авторизации и белое окно после успешного входа, удаляя всю сложную навигацию, компоненты UI и связанные спецификации.

## Глоссарий

- **Auth Gate**: Компонент авторизации, отображающий экран входа через Google OAuth
- **White Window**: Пустое белое окно без контента, отображаемое после успешной авторизации
- **UI Components**: React компоненты пользовательского интерфейса (navigation, dashboard, calendar, tasks, contacts, settings, meeting-detail, logo)
- **Sidebar**: Боковая панель навигации с возможностью сворачивания
- **IPC Handlers**: Обработчики межпроцессного взаимодействия в Electron
- **Functional Tests**: Автоматизированные тесты UI с использованием Playwright
- **Spec Documents**: Документы спецификаций в директории .kiro/specs/

## Требования

### Требование 1: Упрощение Главного Компонента

**User Story:** Как разработчик, я хочу упростить главный компонент приложения, чтобы он показывал только авторизацию и белое окно.

#### Критерии Приемки

1. КОГДА пользователь не авторизован, ТО "App Component" ДОЛЖЕН отображать только "Auth Gate"
2. КОГДА пользователь успешно авторизован, ТО "App Component" ДОЛЖЕН отображать белое окно без какого-либо контента
3. "App Component" НЕ ДОЛЖЕН содержать логику навигации между экранами
4. "App Component" НЕ ДОЛЖЕН содержать состояние для текущего экрана или выбранных элементов
5. "App Component" НЕ ДОЛЖЕН содержать логику управления состоянием сайдбара

**User Story:** Как разработчик, я хочу удалить все UI компоненты кроме auth-gate, чтобы упростить кодовую базу.

#### Критерии Приемки

2. "Dashboard Component" ДОЛЖЕН быть удален из кодовой базы
3. "Calendar Component" ДОЛЖЕН быть удален из кодовой базы
4. "Tasks Component" ДОЛЖЕН быть удален из кодовой базы
5. "Contacts Component" ДОЛЖЕН быть удален из кодовой базы
6. "Settings Component" ДОЛЖЕН быть удален из кодовой базы
7. "Meeting Detail Component" ДОЛЖЕН быть удален из кодовой базы
8. "Logo Component" ДОЛЖЕН быть удален из кодовой базы
9. "Status Badge Component" ДОЛЖЕН быть удален из кодовой базы

### Требование 3: Удаление IPC Обработчиков Сайдбара

**User Story:** Как разработчик, я хочу удалить IPC обработчики для сайдбара, так как они больше не нужны.

#### Критерии Приемки

1. КОГДА приложение запускается, ТО IPC обработчик "sidebar:get-state" НЕ ДОЛЖЕН быть зарегистрирован
2. КОГДА приложение запускается, ТО IPC обработчик "sidebar:set-state" НЕ ДОЛЖЕН быть зарегистрирован
3. Файл с обработчиками сайдбара ДОЛЖЕН быть удален из кодовой базы
4. Типы IPC для сайдбара ДОЛЖНЫ быть удалены из определений типов

### Требование 5: Удаление Модульных Тестов UI

**User Story:** Как разработчик, я хочу удалить модульные тесты для удаленных компонентов, чтобы избежать ошибок сборки.

#### Критерии Приемки

1. Тест "sidebar-ipc-handlers.test.ts" ДОЛЖЕН быть удален
2. Тест "sidebar-persistence.test.ts" ДОЛЖЕН быть удален
3. КОГДА запускается команда тестирования, ТО НЕ ДОЛЖНО быть ошибок о несуществующих модулях

### Требование 6: Удаление Спецификаций UI

**User Story:** Как разработчик, я хочу удалить спецификации для функциональности UI, которая больше не существует.

#### Критерии Приемки

1. Спецификация "sidebar-navigation" ДОЛЖНА быть удалена (requirements.md, design.md, tasks.md)
2. Спецификация "branding-system" ДОЛЖНА быть удалена (requirements.md, design.md, tasks.md)
3. Директории спецификаций ДОЛЖНЫ быть полностью удалены

### Требование 7: Сохранение Базовой Инфраструктуры

**User Story:** Как разработчик, я хочу сохранить базовую инфраструктуру приложения, чтобы платформа продолжала работать.

#### Критерии Приемки

1. Спецификация "platform-foundation" ДОЛЖНА остаться без изменений
2. Спецификация "data-storage" ДОЛЖНА остаться без изменений
3. Спецификация "google-oauth-auth" ДОЛЖНА остаться без изменений
4. Спецификация "testing-infrastructure" ДОЛЖНА остаться без изменений
5. Функциональные тесты авторизации ДОЛЖНЫ остаться без изменений
6. IPC обработчики авторизации ДОЛЖНЫ остаться без изменений

### Требование 8: Обновление Типов IPC

**User Story:** Как разработчик, я хочу обновить определения типов IPC, чтобы они отражали только оставшуюся функциональность.

#### Критерии Приемки

1. КОГДА определения типов IPC обновлены, ТО они ДОЛЖНЫ содержать только методы авторизации
2. Типы для методов сайдбара ДОЛЖНЫ быть удалены из интерфейса "ClerklyAPI"
3. КОГДА TypeScript компилируется, ТО НЕ ДОЛЖНО быть ошибок типов

### Требование 9: Белое Окно После Авторизации

**User Story:** Как пользователь, я хочу видеть простое белое окно после авторизации, чтобы интерфейс был минималистичным.

#### Критерии Приемки

1. КОГДА пользователь успешно авторизован, ТО окно ДОЛЖНО отображать белый фон
2. КОГДА пользователь успешно авторизован, ТО окно НЕ ДОЛЖНО содержать никаких UI элементов кроме белого фона
3. Белое окно ДОЛЖНО занимать всю доступную область окна приложения

### Требование 10: Сохранение Функциональности Авторизации

**User Story:** Как пользователь, я хочу иметь возможность войти и выйти из системы, чтобы управлять своей сессией.

#### Критерии Приемки

1. КОГДА пользователь не авторизован, ТО ДОЛЖЕН отображаться экран авторизации
2. КОГДА пользователь нажимает кнопку входа, ТО ДОЛЖЕН запускаться процесс Google OAuth
3. КОГДА авторизация успешна, ТО приложение ДОЛЖНО переключиться на белое окно
4. КОГДА авторизация не удалась, ТО ДОЛЖНО отображаться сообщение об ошибке
5. Функциональность выхода из системы ДОЛЖНА быть доступна (через горячие клавиши или другой механизм)
