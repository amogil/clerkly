# Документ Требований: Error Notifications

## Введение

Данный документ описывает требования к системе уведомлений об ошибках в приложении Clerkly, включая отображение ошибок фоновых процессов, логирование и автоматическое закрытие уведомлений.

## Глоссарий

- **Error Notification** - Уведомление об ошибке, отображаемое пользователю
- **Toast** - Всплывающее уведомление, автоматически исчезающее через определенное время
- **Background Process** - Фоновый процесс (загрузка данных, синхронизация, API запрос)
- **Logger** - Централизованный класс для логирования событий и ошибок (см. clerkly.3)
- **Error Context** - Контекст ошибки (какая операция вызвала ошибку)
- **IPC Event** - Событие межпроцессного взаимодействия между Main Process и Renderer Process

## Требования

### 1. Обработка Ошибок

**ID:** error-notifications.1

**User Story:** Как пользователь, я хочу видеть понятные сообщения об ошибках, чтобы понимать что пошло не так и что делать дальше.

**Зависимости:** clerkly.3 (Logger)

#### Критерии Приемки

1.1. КОГДА происходит ошибка в фоновом процессе (загрузка данных, синхронизация, API запрос), ТО приложение ДОЛЖНО показать уведомление об ошибке пользователю

1.2. Уведомление об ошибке ДОЛЖНО содержать:
   - Краткое описание проблемы
   - Контекст операции (что пыталось выполниться)

1.3. Уведомление об ошибке ДОЛЖНО автоматически исчезать через 15 секунд ИЛИ при клике пользователя

1.4. Приложение ДОЛЖНО логировать все ошибки через централизованный "Logger" класс (см. clerkly.3) для отладки

1.5. **Исключения для Race Conditions**: Следующие ошибки НЕ ДОЛЖНЫ показываться пользователю (только логироваться):
   - "No user logged in" во время logout (см. user-data-isolation.1.21)
   - Ошибки операций, которые были отменены системой
   - Ошибки, возникшие из-за одновременных операций (race conditions)

**Примечание:** Race conditions - это ситуации, когда операции выполняются одновременно и могут конфликтовать. Такие ошибки являются ожидаемыми и не требуют уведомления пользователя, но должны логироваться для отладки.

**Тестируемость:** Да - через модульные тесты "ErrorNotificationManager" и "NotificationUI" компонентов, функциональные тесты отображения уведомлений и автоматического закрытия

#### Функциональные Тесты

- `tests/functional/error-notifications.spec.ts` - "should show error notification on background process failure"
- `tests/functional/error-notifications.spec.ts` - "should display error context and description"
- `tests/functional/error-notifications.spec.ts` - "should auto-dismiss error notification after 15 seconds"
- `tests/functional/error-notifications.spec.ts` - "should log errors to console"

### 2. Автоматическая обработка IPC ошибок

**ID:** error-notifications.2

**User Story:** Как разработчик, я хочу чтобы ошибки IPC вызовов автоматически показывались пользователю, чтобы не писать обработку ошибок в каждом хуке.

**Зависимости:** error-notifications.1 (обработка ошибок)

#### Критерии Приемки

2.1. Приложение ДОЛЖНО предоставлять wrapper функцию `callApi()` для IPC вызовов, которая автоматически обрабатывает ошибки

2.2. `callApi()` ДОЛЖНА принимать параметры:
   - `apiCall` - функция IPC вызова
   - `context` - контекст операции (строка)
   - `options.silent` - опциональный флаг для отключения toast уведомлений

2.3. КОГДА IPC вызов возвращает `success: false`, ТО `callApi()` ДОЛЖНА показать toast с ошибкой и контекстом

2.4. КОГДА IPC вызов выбрасывает исключение, ТО `callApi()` ДОЛЖНА показать toast с ошибкой и контекстом

2.5. `callApi()` ДОЛЖНА возвращать `data` при успехе или `null` при ошибке

2.6. КОГДА `options.silent = true`, ТО toast уведомления НЕ ДОЛЖНЫ показываться (для специальных случаев)

2.7. Приложение ДОЛЖНО иметь global unhandled rejection handler для необработанных промисов

2.8. Global handler ДОЛЖЕН показывать toast с сообщением "Unexpected error" и логировать ошибку

**Тестируемость:** Да - через модульные тесты `callApi()` функции и интеграционные тесты с хуками

#### Функциональные Тесты

- `tests/unit/utils/apiWrapper.test.ts` - "should show toast on IPC error"
- `tests/unit/utils/apiWrapper.test.ts` - "should return null on error"
- `tests/unit/utils/apiWrapper.test.ts` - "should not show toast when silent is true"
- `tests/unit/utils/apiWrapper.test.ts` - "should return data on success"

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Кастомизация внешнего вида уведомлений (цвет, размер, позиция)
- Уведомления об успешных операциях (success notifications)
- История уведомлений
- Настройка времени автоматического закрытия уведомлений
- Звуковые уведомления
