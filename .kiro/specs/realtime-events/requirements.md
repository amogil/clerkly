# Документ Требований: Real-time Events System

## Введение

Данный документ описывает требования к системе событий реального времени (Real-time Events System) для приложения Clerkly. Система обеспечивает синхронизацию данных между различными частями приложения, вкладками/окнами и автоматическое обновление UI при изменениях в базе данных.

## Глоссарий

- **Actor (Актор)** - Компонент или процесс, который записывает данные в БД и генерирует события об изменениях
- **Consumer (Консюмер)** - Компонент, который подписывается на события и обновляет UI или выполняет другие действия
- **Event (Событие)** - Уведомление об изменении данных, содержащее тип изменения и данные
- **Event Bus** - Центральный механизм для публикации и подписки на события
- **Event Type** - Тип события (agent.created, agent.updated, message.created и т.д.)
- **Event Payload** - Данные, передаваемые вместе с событием
- **Subscription (Подписка)** - Регистрация консюмера на получение событий определенного типа

## Архитектурный Принцип

Система построена на паттерне Publisher-Subscriber (Pub/Sub):
- Акторы публикуют события после записи в БД
- Консюмеры подписываются на интересующие их типы событий
- Event Bus обеспечивает доставку событий от акторов к консюмерам
- События могут содержать полные данные (по умолчанию) или только ID для последующей загрузки из БД

## Требования

### 1. Публикация событий

**ID:** realtime-events.1

**User Story:** Как актор, я хочу публиковать события об изменениях данных, чтобы другие части приложения могли на них реагировать.

#### Критерии Приемки

1.1. Актор ДОЛЖЕН иметь возможность опубликовать событие с указанием типа и данных

1.2. Публикация события ДОЛЖНА происходить ПОСЛЕ успешной записи в БД

1.3. ЕСЛИ запись в БД не удалась, ТО событие НЕ ДОЛЖНО публиковаться

1.4. Событие ДОЛЖНО содержать:
   - `type: string` - тип события (например, "agent.created")
   - `payload: object` - данные события
   - `timestamp: Date` - время генерации события
   - `source: string` - идентификатор источника события

1.5. По умолчанию событие ДОЛЖНО содержать полные данные измененной сущности

1.6. Актор МОЖЕТ опубликовать событие только с ID сущности (для больших объектов)

1.7. Публикация события ДОЛЖНА быть асинхронной и не блокировать основной поток

- `tests/functional/realtime-events.spec.ts` - "should publish event after successful DB write"
- `tests/functional/realtime-events.spec.ts` - "should not publish event if DB write fails"
- `tests/functional/realtime-events.spec.ts` - "should include full data in event by default"

### 2. Подписка на события

**ID:** realtime-events.2

**User Story:** Как консюмер, я хочу подписаться на события определенного типа, чтобы реагировать на изменения данных.

#### Критерии Приемки

2.1. Консюмер ДОЛЖЕН иметь возможность подписаться на события по типу

2.2. Консюмер ДОЛЖЕН иметь возможность подписаться на несколько типов событий одновременно

2.3. Консюмер ДОЛЖЕН иметь возможность подписаться на все события (wildcard "*")

2.4. Подписка ДОЛЖНА возвращать функцию для отписки

2.5. При получении события консюмер ДОЛЖЕН получить callback с данными события

2.6. Callback ДОЛЖЕН выполняться асинхронно

2.7. ЕСЛИ callback выбрасывает исключение, ТО оно НЕ ДОЛЖНО влиять на другие консюмеры

2.8. Консюмер ДОЛЖЕН автоматически отписываться при unmount компонента (для React)

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should receive events after subscription"
- `tests/functional/realtime-events.spec.ts` - "should support multiple event types subscription"
- `tests/functional/realtime-events.spec.ts` - "should unsubscribe correctly"
- `tests/functional/realtime-events.spec.ts` - "should isolate consumer errors"

### 3. Типы событий для агентов

**ID:** realtime-events.3

**User Story:** Как разработчик, я хочу иметь стандартизированные типы событий для агентов, чтобы обеспечить консистентность системы.

#### Критерии Приемки

3.1. Система ДОЛЖНА поддерживать следующие типы событий для агентов:
   - `agent.created` - создан новый агент
   - `agent.updated` - обновлены данные агента (title, description, status, updatedAt)
   - `agent.deleted` - агент удален
   - `agent.status.changed` - изменился статус агента

3.2. Событие `agent.created` ДОЛЖНО содержать полные данные нового агента

3.3. Событие `agent.updated` ДОЛЖНО содержать:
   - Полные данные агента после обновления
   - Список измененных полей

3.4. Событие `agent.status.changed` ДОЛЖНО содержать:
   - ID агента
   - Старый статус
   - Новый статус
   - Время изменения

3.5. Событие `agent.deleted` ДОЛЖНО содержать только ID удаленного агента

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should emit agent.created with full data"
- `tests/functional/realtime-events.spec.ts` - "should emit agent.updated with changed fields"
- `tests/functional/realtime-events.spec.ts` - "should emit agent.status.changed"

### 4. Типы событий для сообщений

**ID:** realtime-events.4

**User Story:** Как разработчик, я хочу иметь стандартизированные типы событий для сообщений, чтобы обеспечить консистентность системы.

#### Критерии Приемки

4.1. Система ДОЛЖНА поддерживать следующие типы событий для сообщений:
   - `message.created` - создано новое сообщение
   - `message.updated` - обновлено сообщение
   - `message.deleted` - сообщение удалено

4.2. Событие `message.created` ДОЛЖНО содержать:
   - Полные данные сообщения

4.3. Событие `message.updated` ДОЛЖНО содержать:
   - Полные данные сообщения после обновления
   - ID агента
   - Список измененных полей

4.4. Событие `message.deleted` ДОЛЖНО содержать:
   - ID удаленного сообщения
   - ID агента

### 5. Синхронизация между вкладками/окнами

**ID:** realtime-events.5

**User Story:** Как пользователь, я хочу видеть изменения данных во всех открытых вкладках/окнах приложения в реальном времени.

#### Критерии Приемки

5.1. События ДОЛЖНЫ передаваться между всеми открытыми вкладками/окнами приложения

5.2. Для передачи событий между вкладками ДОЛЖЕН использоваться BroadcastChannel API

5.3. ЕСЛИ BroadcastChannel не поддерживается, ТО ДОЛЖЕН использоваться fallback через localStorage events

5.4. События из других вкладок ДОЛЖНЫ обрабатываться так же, как локальные события

5.5. Вкладка НЕ ДОЛЖНА получать события, которые она сама опубликовала (избежать дублирования)

5.6. При закрытии вкладки все её подписки ДОЛЖНЫ автоматически очищаться

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should sync events between tabs"
- `tests/functional/realtime-events.spec.ts` - "should not receive own events"
- `tests/functional/realtime-events.spec.ts` - "should cleanup on tab close"

### 6. Обработка событий в UI

**ID:** realtime-events.6

**User Story:** Как пользователь, я хочу видеть изменения данных в UI без необходимости обновлять страницу.

#### Критерии Приемки

6.1. КОГДА приходит событие `agent.created`, ТО новый агент ДОЛЖЕН добавляться в список

6.2. КОГДА приходит событие `agent.updated`, ТО данные агента в UI ДОЛЖНЫ обновляться

6.3. КОГДА приходит событие `agent.status.changed`, ТО визуальный индикатор статуса ДОЛЖЕН обновляться

6.4. КОГДА приходит событие `message.created`, ТО:
   - Сообщение ДОЛЖНО добавляться в чат соответствующего агента
   - updatedAt агента ДОЛЖЕН обновляться
   - ЕСЛИ пользователь внизу чата, ТО ДОЛЖЕН выполняться автоскролл

6.5. Обновления UI ДОЛЖНЫ происходить плавно без мерцания

6.6. ЕСЛИ пользователь редактирует данные, ТО входящие события НЕ ДОЛЖНЫ перезаписывать его изменения

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should add new agent to list on agent.created"
- `tests/functional/realtime-events.spec.ts` - "should update agent data on agent.updated"
- `tests/functional/realtime-events.spec.ts` - "should add message to chat on message.created"
- `tests/functional/realtime-events.spec.ts` - "should not overwrite user edits"

### 7. Производительность и оптимизация

**ID:** realtime-events.7

**User Story:** Как разработчик, я хочу, чтобы система событий работала эффективно и не создавала проблем с производительностью.

#### Критерии Приемки

7.1. Обработка события ДОЛЖНА занимать < 10ms

7.2. Система ДОЛЖНА поддерживать минимум 100 событий в секунду без деградации производительности

7.3. ЕСЛИ событий приходит слишком много, ТО ДОЛЖНА применяться throttling/debouncing стратегия

7.4. События одного типа для одной сущности ДОЛЖНЫ объединяться (batching)

7.5. Старые неиспользуемые подписки ДОЛЖНЫ автоматически очищаться

7.6. Система ДОЛЖНА логировать метрики производительности (количество событий, время обработки)

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should handle 100 events per second"
- `tests/functional/realtime-events.spec.ts` - "should batch events for same entity"
- `tests/functional/realtime-events.spec.ts` - "should cleanup unused subscriptions"

## Нефункциональные Требования

### Надежность

- События НЕ ДОЛЖНЫ теряться при нормальной работе приложения
- ЕСЛИ консюмер недоступен, событие ДОЛЖНО быть пропущено (at-most-once delivery)
- Система ДОЛЖНА корректно работать при сетевых проблемах
- Ошибки в одном консюмере НЕ ДОЛЖНЫ влиять на других консюмеров

### Производительность

- Публикация события: < 1ms
- Доставка события консюмеру: < 10ms
- Поддержка минимум 100 событий/сек
- Память: < 10MB для Event Bus

### Масштабируемость

- Поддержка до 1000 активных подписок одновременно
- Поддержка до 100 типов событий
- Поддержка до 10 открытых вкладок/окон

### Отладка и мониторинг

- Все события ДОЛЖНЫ логироваться в development режиме
- Система ДОЛЖНА предоставлять метрики:
  - Количество опубликованных событий по типам
  - Количество активных подписок
  - Время обработки событий
  - Количество ошибок в консюмерах

## Зависимости

### Внешние библиотеки
- Нет (используются только нативные Web APIs)

### Browser APIs
- `BroadcastChannel` - для синхронизации между вкладками
- `localStorage` - fallback для синхронизации между вкладками
- `CustomEvent` - для внутренних событий

### Внутренние компоненты
- `Logger` - для логирования событий и ошибок
- `DataManager` - для работы с БД

## Ограничения

### Текущая реализация

- Нет персистентности событий (события существуют только в памяти)
- Нет гарантии доставки (at-most-once)
- Нет поддержки offline режима
- Нет replay событий

### Будущие улучшения

- Event sourcing для полной истории изменений
- Персистентность событий в БД
- Replay событий для восстановления состояния
- Поддержка offline режима с синхронизацией при подключении
- WebSocket для real-time обновлений с сервера

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Синхронизация с сервером (только локальная синхронизация)
- Конфликт-резолюция при одновременном редактировании
- Транзакции и откат изменений
- Шифрование событий
- Аутентификация и авторизация событий
