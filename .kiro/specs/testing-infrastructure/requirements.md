# Документ Требований: Инфраструктура Тестирования

## Введение

Данный документ описывает требования к инфраструктуре тестирования приложения Clerkly, включая типы тестов, стратегию мокирования и процесс валидации.

## Глоссарий

- **Unit Tests** - Модульные тесты, проверяющие отдельные функции/классы изолированно
- **Property-Based Tests** - Тесты на основе свойств, проверяющие инварианты на множестве входных данных
- **Integration Tests** - Интеграционные тесты, проверяющие взаимодействие компонентов с реальным Electron
- **Functional Tests** - Функциональные тесты, проверяющие end-to-end сценарии с реальным Electron
- **Mock** - Имитация внешней зависимости для изоляции тестируемого кода
- **Real Electron** - Запуск реального Electron процесса в тестовой среде

## Требования

### 1. Стратегия Мокирования в Модульных Тестах

**ID:** testing.1

**User Story:** Как разработчик, я хочу чтобы модульные тесты были быстрыми и изолированными, чтобы быстро получать обратную связь о качестве кода.

**Зависимости:** Нет

#### Критерии Приемки

1.1. WHEN модульный тест выполняется, ALL внешние зависимости SHALL быть замокированы

1.2. THE следующие зависимости SHALL быть замокированы в модульных тестах:
- Electron API (`BrowserWindow`, `app`, `screen`, `ipcMain`)
- Внешние модули (`fs`, `path`, `crypto`)
- Зависимости класса (например, `DataManager`)
- Сетевые запросы (`fetch`, `https`)
- База данных (SQLite)

1.3. EACH модульный тест SHALL выполняться менее чем за 100ms

1.4. EACH модульный тест SHALL быть полностью изолирован от других тестов

### 2. Стратегия Мокирования в Property-Based Тестах

**ID:** testing.2

**User Story:** Как разработчик, я хочу чтобы property-based тесты проверяли инварианты на множестве входных данных, чтобы находить граничные случаи.

**Зависимости:** testing.1

#### Критерии Приемки

2.1. WHEN property-based тест выполняется, ALL внешние зависимости SHALL быть замокированы (аналогично модульным тестам)

2.2. EACH property-based тест SHALL использовать генераторы случайных данных (fast-check)

2.3. EACH property-based тест SHALL выполнять минимум 100 итераций

2.4. EACH property-based тест SHALL проверять инварианты, а не конкретные значения

### 3. Использование Реального Electron в Интеграционных Тестах

**ID:** testing.3

**User Story:** Как разработчик, я хочу чтобы интеграционные тесты использовали реальный Electron, чтобы проверять реальное поведение приложения.

**Зависимости:** Нет

#### Критерии Приемки

3.1. WHEN интеграционный тест выполняется, THE реальный Electron SHALL быть запущен

3.2. THE интеграционные тесты SHALL НЕ мокировать Electron API

3.3. THE интеграционные тесты SHALL использовать реальные классы приложения

3.4. THE интеграционные тесты SHALL использовать реальную базу данных (SQLite в временной директории)

3.5. THE интеграционные тесты SHALL использовать реальную файловую систему

3.6. EACH интеграционный тест SHALL очищать временные файлы после выполнения

### 4. Использование Реального Electron в Функциональных Тестах

**ID:** testing.4

**User Story:** Как разработчик, я хочу чтобы функциональные тесты проверяли end-to-end сценарии с реальным Electron, чтобы убедиться в корректности работы приложения.

**Зависимости:** testing.3

#### Критерии Приемки

4.1. WHEN функциональный тест выполняется, THE реальный Electron SHALL быть запущен

4.2. THE функциональные тесты SHALL НЕ мокировать Electron API

4.3. THE функциональные тесты SHALL использовать реальные классы приложения

4.4. THE функциональные тесты SHALL использовать реальную базу данных (SQLite в временной директории)

4.5. THE функциональные тесты SHALL использовать реальную файловую систему

4.6. THE функциональные тесты SHALL показывать реальные окна на экране

4.7. EACH функциональный тест SHALL очищать временные файлы после выполнения

### 5. Процесс Валидации

**ID:** testing.5

**User Story:** Как разработчик, я хочу чтобы процесс валидации был быстрым и не мешал работе, чтобы часто проверять качество кода.

**Зависимости:** testing.1, testing.2

#### Критерии Приемки

5.1. WHEN команда `npm run validate` выполняется, THE следующие проверки SHALL быть выполнены:
- TypeScript компиляция
- ESLint проверка
- Prettier проверка
- Модульные тесты
- Property-based тесты
- Проверка покрытия кода

5.2. THE команда `npm run validate` SHALL НЕ запускать интеграционные тесты

5.3. THE команда `npm run validate` SHALL НЕ запускать функциональные тесты

5.4. THE процесс валидации SHALL завершаться менее чем за 30 секунд

5.5. WHEN валидация проваливается, THE скрипт SHALL возвращать ненулевой код выхода

### 6. Запуск Интеграционных и Функциональных Тестов

**ID:** testing.6

**User Story:** Как разработчик, я хочу иметь возможность запускать интеграционные и функциональные тесты отдельно, чтобы проверять реальное поведение приложения.

**Зависимости:** testing.3, testing.4

#### Критерии Приемки

6.1. THE команда `npm run test:integration` SHALL запускать только интеграционные тесты

6.2. THE команда `npm run test:functional` SHALL запускать только функциональные тесты

6.3. THE интеграционные и функциональные тесты SHALL запускаться только по явному запросу разработчика

6.4. THE интеграционные и функциональные тесты SHALL НЕ запускаться автоматически в CI/CD

6.5. WHEN интеграционные или функциональные тесты запускаются, THE пользователь SHALL быть предупрежден о том, что будут показаны окна на экране

### 7. Требования к Окружению для Интеграционных и Функциональных Тестов

**ID:** testing.7

**User Story:** Как разработчик, я хочу чтобы интеграционные и функциональные тесты работали в любом окружении, чтобы не зависеть от конкретной конфигурации.

**Зависимости:** testing.3, testing.4

#### Критерии Приемки

7.1. THE интеграционные и функциональные тесты SHALL работать на macOS с графической средой

7.2. THE интеграционные и функциональные тесты SHALL работать на Linux с X11 или Wayland

7.3. THE интеграционные и функциональные тесты SHALL работать на Windows с графической средой

7.4. WHEN графическая среда недоступна, THE тесты SHALL пропускаться с предупреждением

7.5. THE интеграционные и функциональные тесты SHALL использовать временные директории для данных

### 8. Разделение Production Code и Reference Code

**ID:** testing.8

**User Story:** Как разработчик, я хочу четко разделять production код и референсный код, чтобы не тратить время на тестирование и анализ прототипов.

**Зависимости:** Нет

#### Критерии Приемки

8.1. THE код в директории `src/` SHALL считаться Production Code

8.2. THE Production Code SHALL быть покрыт тестами согласно требованиям testing.1-testing.4

8.3. THE Production Code SHALL анализироваться линтерами и статическими анализаторами

8.4. THE Production Code SHALL поддерживаться и обновляться

8.5. THE код в директории `figma/` SHALL считаться Reference Code

8.6. THE Reference Code SHALL НЕ требовать покрытия тестами

8.7. THE Reference Code SHALL НЕ анализироваться линтерами (может быть исключен из конфигурации)

8.8. THE Reference Code SHALL НЕ требовать изменений или поддержки

8.9. THE Reference Code SHALL использоваться только как референс для визуального дизайна и прототипирования

8.10. WHEN измеряется покрытие кода тестами, THE Reference Code SHALL быть исключен из расчетов

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- E2E тесты с Playwright (могут быть добавлены позже)
- Визуальное регрессионное тестирование
- Тесты производительности
- Нагрузочное тестирование
- Тестирование безопасности
