# Документ Требований: UI Приложения

## Введение

Данный документ описывает требования к пользовательскому интерфейсу приложения Clerkly, включая конфигурацию главного окна, поведение при запуске и визуальное оформление.

## Глоссарий

- **Main Window** - Главное окно приложения
- **Maximized State** - Развернутое состояние окна (занимает весь экран, но не полноэкранный режим)
- **Fullscreen Mode** - Полноэкранный режим (скрывает системные элементы macOS)
- **Title Bar** - Заголовок окна приложения
- **Window State** - Состояние окна (позиция, размер, развернутое состояние)
- **Window Bounds** - Границы окна (x, y, width, height)
- **Database** - Локальная база данных SQLite, управляемая через DataManager
- **UI Component** - Компонент пользовательского интерфейса (React компонент)
- **Access Token** - Токен доступа OAuth, используемый для авторизации API запросов (срок действия ~1 час)
- **Refresh Token** - Токен обновления OAuth, используемый для получения нового access token без повторной авторизации пользователя
- **Token Expiry** - Истечение срока действия токена (expires_in)
- **HTTP 401 Unauthorized** - Ошибка авторизации, указывающая на невалидный или истекший токен
- **Session Expiry** - Истечение сессии пользователя, требующее повторной авторизации
- **AI Agent Settings** - Настройки AI агента (LLM провайдер и API ключ)
- **LLM Provider** - Провайдер языковой модели (OpenAI, Anthropic, Google)
- **API Key** - Ключ доступа к API провайдера LLM
- **System Locale** - Системная локаль операционной системы, определяющая формат дат и времени
- **Debounce** - Задержка выполнения операции до окончания серии быстрых изменений
- **User Email** - Email адрес пользователя из Google OAuth, используемый для изоляции данных
- **Data Isolation** - Изоляция данных между пользователями по email

## Архитектурные Принципы

### Принцип Единого Источника Истины (Single Source of Truth)

Приложение следует архитектурному принципу, где **база данных является единственным источником истины** для всех данных приложения.

**Ключевые правила:**

1. **UI отображает данные из базы данных**: Все компоненты пользовательского интерфейса читают и отображают данные непосредственно из локальной базы данных (SQLite через DataManager). UI не хранит собственное состояние данных, кроме временного состояния загрузки/ошибок.

2. **Автоматическое обновление UI при изменении данных**: Когда данные в базе данных изменяются, UI автоматически обновляется для отражения новых данных. Это достигается через механизм событий (IPC events) между Main Process и Renderer Process.

3. **Фоновые процессы обновляют базу данных**: Фоновые процессы (например, синхронизация с внешними API, обновление токенов, периодические задачи) изменяют данные непосредственно в базе данных. После изменения данных они отправляют события в Renderer Process для обновления UI.

**Преимущества этого подхода:**

- **Консистентность данных**: Все части приложения видят одни и те же данные
- **Offline-first**: Приложение работает с локальными данными, синхронизация происходит в фоне
- **Плавный UX**: Нет мерцания пустых состояний, UI всегда показывает актуальные данные из базы
- **Простота отладки**: Легко проверить состояние данных, просто посмотрев в базу данных
- **Надежность**: Данные персистентны, не теряются при перезапуске приложения

**Пример потока данных:**

```
┌─────────────────────────────────────────────────────────┐
│                   Main Process                          │
│                                                          │
│  ┌──────────────┐      ┌──────────────┐                │
│  │   External   │─────▶│  DataManager │                │
│  │   API Call   │      │   (SQLite)   │                │
│  └──────────────┘      └──────┬───────┘                │
│                               │                          │
│                               │ Data Changed             │
│                               ▼                          │
│                        ┌──────────────┐                 │
│                        │  IPC Event   │                 │
│                        │  Emitter     │                 │
│                        └──────┬───────┘                 │
└───────────────────────────────┼─────────────────────────┘
                                │
                                │ Event: 'data:updated'
                                ▼
┌─────────────────────────────────────────────────────────┐
│                 Renderer Process                         │
│                                                          │
│  ┌──────────────┐      ┌──────────────┐                │
│  │  IPC Event   │─────▶│ UI Component │                │
│  │  Listener    │      │  (React)     │                │
│  └──────────────┘      └──────┬───────┘                │
│                               │                          │
│                               │ Request Fresh Data       │
│                               ▼                          │
│                        ┌──────────────┐                 │
│                        │  IPC Call    │                 │
│                        │  to Main     │                 │
│                        └──────────────┘                 │
└─────────────────────────────────────────────────────────┘
```

### Принцип Управления Токенами и Авторизацией

Приложение следует строгим правилам управления токенами авторизации и обработки ошибок авторизации для обеспечения безопасности и плавного пользовательского опыта.

**Ключевые правила:**

1. **Автоматическое обновление токенов**: Когда access token истекает (expires_in), система автоматически обновляет его через refresh token без участия пользователя. Это происходит в фоновом режиме через `OAuthClientManager.refreshAccessToken()`. Пользователь продолжает работу без прерываний.

2. **Обработка ошибок авторизации (HTTP 401)**: При получении ошибки HTTP 401 Unauthorized от любого API (Google UserInfo, Calendar, Tasks и т.д.), система должна:
   - Немедленно очистить все токены из хранилища
   - Показать экран логина (LoginError компонент с errorCode 'invalid_grant') с сообщением об истечении сессии
   - Пользователь может повторно авторизоваться через кнопку "Continue with Google"
   - **Примечание**: Данные пользователя в базе данных НЕ очищаются - они сохраняются для отображения при следующей авторизации

3. **Централизованная обработка**: Все API запросы должны проходить через централизованный обработчик ошибок, который проверяет статус авторизации (HTTP 401) и выполняет необходимые действия при ошибках авторизации.

4. **Логирование**: Все ошибки авторизации должны логироваться через централизованный "Logger" класс (см. clerkly.3) с контекстом (какой API запрос вызвал ошибку), но пользователю показываются только понятные сообщения без технических деталей.

**Поток обработки ошибки авторизации:**
```
API Request → HTTP 401 → Clear Tokens → Show LoginError Component → Redirect to OAuth
```

**Поток автоматического обновления токена:**
```
Token Expiring → OAuthClientManager.refreshAccessToken() → Update Tokens in Storage → Continue Operation
```

**Преимущества этого подхода:**

- **Безопасность**: Немедленное прекращение доступа при невалидных токенах
- **Прозрачность**: Автоматическое обновление токенов без прерывания работы пользователя
- **Понятность**: Четкое сообщение пользователю о необходимости повторной авторизации
- **Консистентность**: Единый подход к обработке ошибок авторизации во всем приложении
- **Надежность**: Централизованная обработка предотвращает утечки токенов и несогласованное состояние

**ID:** ui.1

1.3. "Main Window" ДОЛЖНО позволять пользователю изменять размер окна (resizable: true)

1.4. "Main Window" ДОЛЖНО позволять пользователю максимизировать окно через стандартные элементы управления macOS

1.5. "Main Window" ДОЛЖНО сохранять видимость системных элементов macOS (меню, dock)

### 6. Блок Account (Профиль Пользователя)

**ID:** ui.6

**User Story:** Как пользователь, я хочу видеть информацию о моем Google аккаунте в приложении.

**Зависимости:** google-oauth-auth.3, google-oauth-auth.5

#### Критерии Приемки

6.1. "Account Block" ДОЛЖЕН отображать следующие поля из профиля пользователя:
   - Имя пользователя (name)
   - Email адрес (email)

6.2. "Account Block" ДОЛЖЕН НЕ позволять пользователю редактировать поля профиля (read-only)

6.3. Система ДОЛЖНА загружать данные профиля из Google UserInfo API:
   - **Синхронно** во время успешной авторизации (после обмена authorization code на токены, во время отображения loader - см. google-oauth-auth.15)
   - При запуске приложения (если пользователь авторизован)
   - При обновлении access token (refresh)

6.4. КОГДА профиль загружается синхронно во время авторизации, ТО:
   - Система ДОЛЖНА показывать loader во время получения профиля
   - При успехе: показать главный интерфейс приложения (Dashboard)
   - При ошибке: очистить токены И показать LoginError компонент с errorCode 'profile_fetch_failed'

6.5. КОГДА профиль не может быть получен во время авторизации, ТО авторизация ДОЛЖНА считаться неуспешной (токены очищаются, пользователь видит LoginError)

**Примечание:** Отображение данных из базы и автоматическое обновление UI при изменении данных следуют из архитектурного принципа. Данные профиля пользователя сохраняются в базе данных даже после logout - очищаются только токены авторизации.

#### Функциональные Тесты

- `tests/functional/account-profile.spec.ts` - "should not allow editing profile fields"
- `tests/functional/account-profile.spec.ts` - "should update profile data when changed in Google"

### 8. Навигация и Авторизация

**ID:** ui.8

**User Story:** Как пользователь, я хочу чтобы приложение автоматически направляло меня на нужный экран в зависимости от статуса авторизации, чтобы не видеть функции, которые мне недоступны.

**Зависимости:** google-oauth-auth.3, google-oauth-auth.11, google-oauth-auth.15

#### Критерии Приемки

8.1. КОГДА пользователь не авторизован, ТО приложение ДОЛЖНО показывать экран логина

8.2. КОГДА пользователь не авторизован, ТО пользователь НЕ МОЖЕТ получить доступ к защищенным экранам (Dashboard, Settings, Tasks, Calendar, Contacts)

8.3. КОГДА пользователь нажимает кнопку "Continue with Google", ТО приложение ДОЛЖНО открыть системный браузер для авторизации

8.4. КОГДА браузер открыт, ТО кнопка "Continue with Google" ДОЛЖНА оставаться активной (пользователь может открыть несколько вкладок браузера)

8.5. КОГДА пользователь завершает авторизацию в браузере И authorization code получен, ТО приложение ДОЛЖНО показать loader с текстом "Signing in..."

8.6. КОГДА loader отображается, ТО приложение ДОЛЖНО выполнить следующие операции:
   - Обменять authorization code на токены
   - Загрузить профиль пользователя из Google UserInfo API (синхронно)

8.7. КОГДА пользователь успешно авторизуется через Google OAuth И профиль загружен, ТО приложение ДОЛЖНО автоматически перенаправить пользователя на Dashboard (главный экран приложения)

8.8. КОГДА происходит ошибка авторизации (обмен токенов ИЛИ загрузка профиля), ТО приложение ДОЛЖНО показать LoginError компонент с описанием ошибки

8.9. КОГДА пользователь выходит из системы (logout), ТО приложение ДОЛЖНО очистить все токены авторизации (см. google-oauth-auth.7) И автоматически перенаправить пользователя на экран логина

**User Story:** Как пользователь, я хочу чтобы приложение автоматически обновляло токены авторизации и корректно обрабатывало ситуации, когда моя сессия истекла, чтобы я мог продолжать работу без прерываний или понимал, что нужно войти заново.

**Зависимости:** google-oauth-auth.3, google-oauth-auth.4

#### Критерии Приемки

9.1. КОГДА access token истекает (expires_in), ТО система ДОЛЖНА автоматически обновить токен через refresh token в фоновом режиме без участия пользователя

9.2. КОГДА автоматическое обновление токена успешно, ТО пользователь ДОЛЖЕН продолжать работу без прерываний и уведомлений

9.3. КОГДА любой API запрос (Google UserInfo, Calendar, Tasks и т.д.) возвращает ошибку HTTP 401 Unauthorized, ТО система ДОЛЖНА:
   - Немедленно очистить все токены из хранилища
   - Показать экран логина (LoginError компонент с errorCode 'invalid_grant')

   - **НЕ** очищать данные пользователя из базы данных (данные сохраняются для отображения при следующей авторизации)

9.4. Система ДОЛЖНА использовать централизованный обработчик для всех API запросов, который проверяет статус авторизации (HTTP 401)

9.5. КОГДА происходит ошибка авторизации (401), ТО система ДОЛЖНА логировать событие с контекстом через "Logger" класс (см. clerkly.3)

9.6. Система ДОЛЖНА показывать пользователю понятное сообщение об истечении сессии на английском языке (errorCode 'invalid_grant' в LoginError компоненте)

### 10. Настройки AI Agent

**ID:** ui.10

**User Story:** Как пользователь, я хочу настроить AI агента для работы с моими задачами, указав провайдера LLM и API ключ, чтобы агент мог выполнять интеллектуальные операции.

**Зависимости:** ui.12 (изоляция данных пользователей)

#### Критерии Приемки

10.1. Страница Settings ДОЛЖНА содержать секцию "AI Agent Settings" с следующими полями:
   - LLM Provider (выпадающий список): OpenAI (GPT), Anthropic (Claude), Google (Gemini)
   - API Key (текстовое поле с возможностью скрыть/показать)

10.2. КОГДА пользователь вводит API ключ, ТО поле ДОЛЖНО отображать символы как пароль (скрыто по умолчанию)

10.3. Поле API Key ДОЛЖНО содержать кнопку показать/скрыть (toggle visibility button) справа внутри поля

10.4. Кнопка показать/скрыть ДОЛЖНА отображать иконку:
   - Иконка "Eye" (открытый глаз) КОГДА API ключ скрыт (type="password")
   - Иконка "EyeOff" (перечеркнутый глаз) КОГДА API ключ виден (type="text")

10.5. КОГДА пользователь кликает на кнопку показать/скрыть, ТО:
   - ЕСЛИ API ключ скрыт: переключить поле в режим text (показать символы) И изменить иконку на "EyeOff"
   - ЕСЛИ API ключ виден: переключить поле в режим password (скрыть символы) И изменить иконку на "Eye"

10.6. Кнопка показать/скрыть ДОЛЖНА быть видна всегда (независимо от того, пустое поле или заполнено)

10.7. Кнопка показать/скрыть НЕ ДОЛЖНА триггерить сохранение данных (только переключает видимость)

10.8. Состояние видимости API ключа (показан/скрыт) НЕ ДОЛЖНО сохраняться между сессиями (всегда начинать со скрытого состояния)

10.9. КОГДА пользователь изменяет API Key, ТО изменения ДОЛЖНЫ автоматически сохраняться в базу данных с debounce 500ms после последнего изменения

10.10. КОГДА пользователь изменяет LLM Provider (выпадающий список), ТО:
   - Изменение провайдера ДОЛЖНО сохраняться немедленно без debounce
   - Поле API Key ДОЛЖНО автоматически загрузить ключ выбранного провайдера (если он был сохранен ранее)
   - ЕСЛИ ключ для выбранного провайдера не найден: поле API Key ДОЛЖНО быть пустым с placeholder "Enter your API key"

10.11. КОГДА пользователь очищает поле API Key (пустая строка), ТО запись для текущего провайдера ДОЛЖНА быть удалена из базы данных

10.12. Система ДОЛЖНА НЕ показывать визуальный индикатор сохранения (сохранение происходит молча в фоне)

10.13. КОГДА сохранение в базу данных не удается, ТО система ДОЛЖНА показать уведомление об ошибке через стандартный механизм обработки ошибок (см. ui.7)

10.14. Система ДОЛЖНА пытаться зашифровать API ключ перед сохранением используя `safeStorage.encryptString()` из Electron

10.15. КОГДА шифрование недоступно (например, на некоторых Linux системах), ТО система ДОЛЖНА сохранить API ключ без шифрования

10.16. Каждый LLM провайдер ДОЛЖЕН иметь отдельное хранилище для своего API ключа:
   - OpenAI: ключи `ai_agent_api_key_openai` и `ai_agent_api_key_openai_encrypted`
   - Anthropic: ключи `ai_agent_api_key_anthropic` и `ai_agent_api_key_anthropic_encrypted`
   - Google: ключи `ai_agent_api_key_google` и `ai_agent_api_key_google_encrypted`

10.17. Система ДОЛЖНА сохранять метаданные о шифровании для каждого провайдера отдельно:
   - Ключ `ai_agent_api_key_{provider}_encrypted`: `true` если зашифрован, `false` если plain text
   - Ключ `ai_agent_api_key_{provider}`: зашифрованная строка или plain text

10.18. LLM Provider ДОЛЖЕН храниться в таблице `user_data` с ключом `ai_agent_llm_provider`

10.19. КОГДА пользователь переключается между провайдерами, ТО:
   - Ключ предыдущего провайдера ДОЛЖЕН сохраниться в базе данных (не удаляется)
   - Ключ нового провайдера ДОЛЖЕН загрузиться из базы данных (если существует)
   - Приложение ДОЛЖНО использовать активного провайдера и его ключ для всех операций

10.20. КОГДА приложение запускается, ТО настройки AI Agent ДОЛЖНЫ загружаться из базы данных текущего пользователя:
   - Загрузить активного провайдера из `ai_agent_llm_provider`
   - Загрузить API ключ активного провайдера из соответствующего ключа
   - Отобразить провайдера и его ключ в UI

10.21. КОГДА настройки не найдены в базе (первый запуск или после очистки), ТО значения по умолчанию ДОЛЖНЫ быть:
   - LLM Provider: `openai`
   - API Key: пустая строка с placeholder "Enter your API key"

10.22. КОГДА API ключ загружается из базы для конкретного провайдера, ТО система ДОЛЖНА:
   - Проверить флаг `ai_agent_api_key_{provider}_encrypted`
   - Если `true`: расшифровать используя `safeStorage.decryptString()`
   - Если `false`: использовать как plain text

10.23. Система НЕ ДОЛЖНА выполнять валидацию формата API ключа

10.24. UI ДОЛЖЕН отображать текст: "Your API key is stored securely. It will only be used to communicate with your selected LLM provider."

10.25. Кнопка "Test Connection" ДОЛЖНА присутствовать в UI, но НЕ ДОЛЖНА быть функциональной (placeholder для будущей функциональности)

10.26. Система ДОЛЖНА использовать существующие IPC каналы для работы с настройками:
   - `save-data` для сохранения LLM провайдера и API ключей
   - `load-data` для загрузки настроек
   - `delete-data` для удаления API ключей

**Примечание:** Изоляция данных AI Agent между пользователями (сохранение и восстановление настроек при смене пользователя) реализуется через ui.12 (добавление колонки `user_email` в таблицу `user_data`). Отдельные требования для изоляции настроек AI Agent не нужны.

#### Функциональные Тесты

- `tests/functional/settings-ai-agent.spec.ts` - "should save API key with debounce"
- `tests/functional/settings-ai-agent.spec.ts` - "should delete API key when field is cleared"

### 12. Изоляция Данных Пользователей

**ID:** ui.12

**User Story:** Как пользователь, я хочу чтобы мои данные (настройки, задачи, контакты) были изолированы от других пользователей, чтобы при входе другого пользователя мои данные оставались приватными и восстанавливались при повторном входе.

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth)

#### Критерии Приемки

12.1. Система ДОЛЖНА определять пользователя по email из Google OAuth профиля

12.2. Таблица `user_data` ДОЛЖНА содержать колонку `user_email TEXT` для привязки данных к пользователю

12.3. КОГДА пользователь сохраняет данные, ТО система ДОЛЖНА автоматически добавлять `user_email` текущего авторизованного пользователя

12.4. КОГДА пользователь загружает данные, ТО система ДОЛЖНА фильтровать данные по `user_email` текущего авторизованного пользователя

12.5. КОГДА пользователь A выходит из системы (logout), ТО его данные ДОЛЖНЫ оставаться в базе данных

12.6. КОГДА пользователь B авторизуется, ТО система ДОЛЖНА показывать только данные пользователя B (фильтрация по `user_email`)

12.7. КОГДА пользователь A снова авторизуется, ТО система ДОЛЖНА восстановить все его данные из базы (фильтрация по `user_email`)

12.8. Изоляция данных ДОЛЖНА применяться ко всем типам данных:
   - Настройки AI Agent (ui.10)
   - Задачи (Tasks)
   - Контакты (Contacts)
   - Календарь (Calendar)
   - Любые другие пользовательские данные

12.9. Таблица `user_data` ДОЛЖНА быть обновлена вручную разработчиком (пересоздание базы):
   - Добавить колонку `user_email TEXT`
   - Создать индекс `idx_user_email` на колонке `user_email`

12.10. DataManager ДОЛЖЕН автоматически получать `user_email` текущего авторизованного пользователя из UserProfileManager

12.11. DataManager ДОЛЖЕН автоматически добавлять `user_email` при сохранении данных (без явного параметра в методе `saveData`)

12.12. DataManager ДОЛЖЕН автоматически фильтровать по `user_email` при загрузке данных (без явного параметра в методе `loadData`)

12.13. КОГДА пользователь не авторизован (нет email), ТО методы DataManager `saveData`/`loadData`/`deleteData` ДОЛЖНЫ возвращать ошибку "No user logged in"

12.14. UserProfileManager ДОЛЖЕН кэшировать email текущего пользователя в памяти (свойство `currentUserEmail`)

12.15. КОГДА пользователь успешно авторизуется через Google OAuth, ТО UserProfileManager ДОЛЖЕН установить `currentUserEmail` из профиля

12.16. КОГДА access token обновляется (refresh), ТО UserProfileManager ДОЛЖЕН обновить `currentUserEmail` из обновленного профиля

12.17. КОГДА приложение запускается И пользователь авторизован (есть валидные токены), ТО UserProfileManager ДОЛЖЕН загрузить профиль из базы данных И установить `currentUserEmail`

12.18. КОГДА пользователь выходит (logout), ТО UserProfileManager ДОЛЖЕН очистить `currentUserEmail` (установить в `null`)

12.19. КОГДА DataManager возвращает ошибку "No user logged in" И пользователь не авторизован, ТО система ДОЛЖНА перенаправить на экран логина И очистить все кэши

12.20. КОГДА DataManager возвращает ошибку "No user logged in" И сессия истекла во время работы, ТО система ДОЛЖНА применить процедуру восстановления сессии (см. ui.9.1, ui.9.3) И при успешном восстановлении повторить операцию с данными без уведомления пользователю

12.21. КОГДА DataManager возвращает ошибку "No user logged in" во время logout (race condition), ТО система ДОЛЖНА молча игнорировать ошибку (логировать через "Logger" класс для отладки, см. clerkly.3)

12.22. Профиль пользователя (ключ `user_profile`) ДОЛЖЕН быть изолирован по email (каждый пользователь имеет свой профиль в базе данных)

12.23. Состояние окна (Window State) ДОЛЖНО быть изолировано по пользователям (каждый пользователь имеет свое положение окна)

12.24. OAuth токены ДОЛЖНЫ быть изолированы по пользователям (каждый пользователь имеет свои токены)

**Примечание:** Эта функциональность критична для multi-user support и должна быть реализована перед ui.10 (настройки AI Agent).

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should isolate data between different users"
- `tests/functional/user-data-isolation.spec.ts` - "should restore user data after re-login"
