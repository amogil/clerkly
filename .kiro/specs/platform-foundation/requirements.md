# Документ Требований - Основа Платформы

## Введение

Этот документ определяет базовые требования к платформе для настольного приложения Clerkly, включая оболочку Electron приложения, инструментарий разработки и архитектуру межпроцессного взаимодействия.

## Глоссарий

- **Clerkly**: Название и бренд настольного приложения
- **Main Process**: Главный процесс Electron, управляющий жизненным циклом приложения
- **Renderer Process**: Процесс рендеринга Electron, обрабатывающий пользовательский интерфейс
- **Preload Script**: Скрипт предзагрузки Electron, обеспечивающий безопасный мост IPC
- **IPC**: Межпроцессное взаимодействие между процессами Electron

## Зависимости

- **Предоставляет**: Оболочку приложения, инфраструктуру IPC, среду разработки
- **Требуется для**: Всех остальных фич (google-oauth-auth, data-storage, sidebar-navigation, branding-system)

## Требования

### Требование 1: Платформа и Оболочка Приложения

**Пользовательская История:** Как пользователь macOS, я хочу запускать Clerkly как нативное настольное приложение, чтобы иметь доступ к нему без веб-браузера.

#### Критерии Приемки

1. "Clerkly Application" ДОЛЖНО быть нацелено на macOS как основную поддерживаемую платформу
2. "Main Process" ДОЛЖЕН инициализироваться с функционирующим процессом рендеринга
3. КОГДА приложение запускается, "Main Window" ДОЛЖНО открываться развернутым, но не в полноэкранном режиме
4. "Application Name" ДОЛЖНО отображать "Clerkly" в меню приложений macOS вместо стандартного имени Electron
5. "Development Scripts" ДОЛЖНЫ предоставлять npm команды для запуска приложения в режиме разработки на macOS

### Требование 2: Инструментарий Разработки и Качество Кода

**Пользовательская История:** Как разработчик, я хочу последовательное качество кода и современные инструменты, чтобы кодовая база была поддерживаемой и следовала лучшим практикам.

#### Критерии Приемки

1. "Code Base" ДОЛЖНА использовать Electron v40.0.0, Node.js v25.5.0 и npm v11.8.0 как требуемые версии инструментария
2. "Application Code" ДОЛЖЕН быть написан полностью на TypeScript
3. "Linting System" ДОЛЖНА использовать ESLint для проверки всех TypeScript и TSX исходников в main, preload, renderer и tests
4. "Code Formatter" ДОЛЖЕН использовать Prettier с правилами форматирования, совместимыми с ESLint
5. "Package Scripts" ДОЛЖНЫ включать команды lint, lint:fix, format и format:check
6. "Linting Scope" ДОЛЖНА выполняться на исходных файлах, файлах конфигурации и тестах под контролем версий

### Требование 3: Межпроцессное Взаимодействие и Архитектура

**Пользовательская История:** Как разработчик, я хочу четкое разделение между слоями приложения, чтобы архитектура была поддерживаемой и безопасной.

#### Критерии Приемки

1. "Main Process", "Preload Script" и "Renderer Process" ДОЛЖНЫ быть разделены на отдельные модули
2. "Cross Layer Imports" ДОЛЖНЫ быть запрещены; взаимодействие между слоями ДОЛЖНО происходить только через IPC
3. "IPC Infrastructure" ДОЛЖНА предоставлять безопасные каналы связи для других фич
4. "IPC Channel Registry" ДОЛЖЕН поддерживать стабильный контракт для межфичного взаимодействия
