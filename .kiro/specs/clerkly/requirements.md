# Документ требований: Clerkly

## Введение

Clerkly - это Electron-приложение для Mac OS X, представляющее собой AI-ассистента для менеджеров. Приложение предоставляет комплексный набор инструментов для управления работой менеджера, включая встречи, документы, задачи, коммуникацию с командой, помощь в принятии решений и генерацию отчетов.

На текущем этапе реализуется базовая платформа приложения с локальным хранением данных, нативным Mac OS X интерфейсом и комплексным тестовым покрытием, создавая надежную основу для будущих AI-функций.

## Цели

- Создать нативное desktop-приложение для Mac OS X на базе Electron
- Обеспечить локальное хранение данных пользователя с использованием SQLite
- Предоставить надежную и протестированную платформу для будущих функций AI-ассистента
- Обеспечить высокую производительность (запуск < 3 секунды, отзывчивый UI)
- Гарантировать безопасность данных пользователя через локальное хранение

## Глоссарий

- **Clerkly**: Electron-приложение AI-ассистента для менеджеров на Mac OS X
- **Пользователь**: Менеджер, использующий приложение
- **Main Process**: Основной процесс Electron, управляющий жизненным циклом приложения и бизнес-логикой
- **Renderer Process**: Процесс отрисовки, отвечающий за UI приложения и взаимодействие с пользователем
- **IPC (Inter-Process Communication)**: Механизм коммуникации между Main Process и Renderer Process
- **SQLite**: Встроенная база данных для локального хранения данных пользователя
- **Property-Based Testing**: Метод тестирования, проверяющий универсальные свойства на множестве сгенерированных входных данных
- **DataManager**: Класс для управления данными в локальной базе данных SQLite
- **Single Source of Truth**: Архитектурный принцип, где база данных является единственным источником истины для всех данных

## Архитектурные Принципы

### Принцип Единого Источника Истины (Single Source of Truth)

Приложение следует фундаментальному архитектурному принципу, где **база данных является единственным источником истины** для всех данных приложения.

**Ключевые правила:**

1. **UI отображает данные из базы данных**: Все компоненты пользовательского интерфейса читают и отображают данные непосредственно из локальной базы данных (SQLite через DataManager). UI не хранит собственное состояние данных, кроме временного состояния загрузки/ошибок.

2. **Автоматическое обновление UI при изменении данных**: Когда данные в базе данных изменяются, UI автоматически обновляется для отражения новых данных. Это достигается через механизм событий (IPC events) между Main Process и Renderer Process.

3. **Фоновые процессы обновляют базу данных**: Фоновые процессы (например, синхронизация с внешними API, обновление токенов, периодические задачи) изменяют данные непосредственно в базе данных. После изменения данных они отправляют события в Renderer Process для обновления UI.

**Преимущества этого подхода:**

- **Консистентность данных**: Все части приложения видят одни и те же данные
- **Offline-first**: Приложение работает с локальными данными, синхронизация происходит в фоне
- **Плавный UX**: Нет мерцания пустых состояний, UI всегда показывает актуальные данные из базы
- **Простота отладки**: Легко проверить состояние данных, просто посмотрев в базу данных
- **Надежность**: Данные персистентны, не теряются при перезапуске приложения

**Применение принципа:**

Этот принцип применяется ко всем типам данных в приложении:
- Состояние окна (window-management)
- Профиль пользователя (account-profile)
- Настройки приложения (settings)
- Данные пользователей с изоляцией (user-data-isolation)
- Токены авторизации (google-oauth-auth)
- Задачи, контакты, календарь (будущие функции)

### Требование 3: Логирование

**ID:** clerkly.3

**User Story:** Как разработчик, я хочу иметь централизованную систему логирования с единообразным форматом, чтобы я мог эффективно отслеживать работу приложения и диагностировать проблемы.

**Зависимости:** clerkly.1

**Обоснование:** Централизованное логирование с единообразным форматом timestamp критически важно для отладки, мониторинга и диагностики проблем в production. Использование фиксированного формата YYYY-MM-DD HH:MM:SS обеспечивает консистентность логов независимо от локали пользователя и упрощает парсинг логов автоматическими инструментами.

#### Критерии приемки

3.1. Clerkly ДОЛЖНО иметь централизованный класс "Logger" для всего логирования в приложении

3.2. "Logger" ДОЛЖЕН использовать "DateTimeFormatter.formatLogTimestamp()" для автоматического форматирования timestamp в логах

3.3. Все логи ДОЛЖНЫ использовать фиксированный формат timestamp: YYYY-MM-DD HH:MM:SS±HH:MM (с указанием часового пояса, независимо от системной локали)

3.4. "Logger" ДОЛЖЕН поддерживать уровни логирования: debug, info, warn, error

3.5. "Logger" ДОЛЖЕН поддерживать два способа использования:
   - Статические методы: `Logger.log(context, message, level)` - для разовых вызовов
   - Параметризованный экземпляр: `Logger.create(context)` - для многократного использования в одном модуле

3.6. При использовании статических методов "Logger" ДОЛЖЕН принимать следующие параметры:
   - context (обязательный): строка с именем компонента
   - message (обязательный): строка с сообщением лога (БЕЗ дублирования контекста)
   - level (опциональный, по умолчанию 'info'): уровень логирования (debug, info, warn, error)

3.7. При использовании параметризованного экземпляра методы ДОЛЖНЫ принимать только message (контекст уже задан при создании)

3.8. "Logger" ДОЛЖЕН автоматически добавлять timestamp к каждому сообщению лога (вызывающий код НЕ передает timestamp)

3.9. Сообщения лога НЕ ДОЛЖНЫ дублировать контекст (например, вместо `Logger.info('Main', '[Main] Starting app')` использовать `logger.info('Starting app')`)

3.10. ВСЕ компоненты приложения ДОЛЖНЫ использовать "Logger" вместо прямых вызовов console.log/console.error/console.warn/console.debug

3.11. ВСЕ тесты (модульные, property-based, функциональные) ДОЛЖНЫ использовать "Logger" для логирования вместо прямых вызовов console.*

3.12. Прямые вызовы console.log/console.error/console.warn/console.debug ЗАПРЕЩЕНЫ в production коде и тестах (за исключением самого класса "Logger" и "DateTimeFormatter")

3.13. "Logger" ДОЛЖЕН быть доступен как в "Main Process", так и в "Renderer Process"

3.14. "Logger" ДОЛЖЕН быть доступен в тестовом окружении (модульные, property-based, функциональные тесты)

3.15. "DateTimeFormatter" НЕ ДОЛЖЕН использовать "Logger" для избежания циклической зависимости (Logger → DateTimeFormatter для форматирования timestamp, поэтому DateTimeFormatter не может использовать Logger)

**Тестируемость:** Да - через модульные тесты "Logger" класса, проверку формата timestamp, property-based тесты для различных входных данных, функциональные тесты генерации логов при навигации

- Тесты удалены после упрощения UI

## Ограничения

### Технические ограничения
- Приложение разрабатывается только для Mac OS X (без поддержки Windows/Linux на текущем этапе)
- Используется TypeScript
- Минимальная версия Node.js: 18+
- Минимальная версия Electron: 28+
- Используется SQLite (better-sqlite3) для локального хранения данных
- Используется Jest для тестирования с fast-check для property-based тестов

### Функциональные ограничения
- На текущем этапе реализуется только базовая структура приложения без AI-функционала
- AI-функции (встречи, документы, задачи, коммуникация, принятие решений, отчеты) будут добавлены в будущих версиях
- Текущая версия фокусируется на платформе, хранении данных, системе миграций и тестировании
- Функциональные тесты запускаются ТОЛЬКО при явной просьбе пользователя (не в автоматическом режиме)

### Ограничения по ресурсам
- Разработка ведется для современных Mac систем (последние 3-4 года)
- Оптимизация для старых систем не является приоритетом

## Риски

### Технические риски

**Риск 1: Проблемы совместимости с разными версиями Mac OS X**
- Вероятность: Средняя
- Влияние: Высокое
- Митигация: Тестирование на разных версиях Mac OS X (10.13+), использование стабильных API Electron

- Вероятность: Низкая

- Митигация: Автоматическое создание backup, восстановление из backup, создание новой базы при повреждении

**Риск 3: Проблемы производительности при больших объемах данных**
- Вероятность: Средняя
- Влияние: Среднее
- Митигация: Оптимизация запросов к SQLite, индексирование, асинхронные операции

### Риски разработки

**Риск 4: Недостаточное тестовое покрытие**
- Вероятность: Низкая
- Влияние: Высокое
- Митигация: Обязательное требование 80%+ покрытия для бизнес-логики и 100% для критических компонентов, property-based тестирование, автоматизация тестов, структурированные комментарии в тестах

**Риск 5: Проблемы с миграциями базы данных**
- Вероятность: Средняя
- Влияние: Высокое
- Митигация: Система миграций с отслеживанием версий, возможность отката миграций, тестирование миграций, backup базы данных перед миграциями
