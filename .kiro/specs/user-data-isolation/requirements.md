# Документ Требований: User Data Isolation

## Введение

Данный документ описывает требования к изоляции данных пользователей в приложении Clerkly, обеспечивая приватность и безопасность данных при использовании приложения несколькими пользователями.

## Глоссарий

- **User Email** - Email адрес пользователя из Google OAuth, используемый для изоляции данных
- **Data Isolation** - Изоляция данных между пользователями по email
- **user_data Table** - Таблица в SQLite для хранения пользовательских данных
- **user_email Column** - Колонка в таблице user_data для привязки данных к пользователю
- **DataManager** - Класс для управления данными в локальной базе данных
- **UserProfileManager** - Класс для управления профилем пользователя и текущим email
- **currentUserEmail** - Кэшированный email текущего авторизованного пользователя
- **Race Condition** - Ситуация, когда операции выполняются одновременно и могут конфликтовать

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `.kiro/specs/clerkly/requirements.md`).

**Применение к User Data Isolation:**
- Все данные пользователей хранятся в базе данных с привязкой к `user_email`
- DataManager автоматически фильтрует данные по текущему пользователю
- При смене пользователя данные предыдущего пользователя остаются в базе и восстанавливаются при повторном входе
- UI автоматически обновляется при смене пользователя, отображая данные нового пользователя из базы

## Требования

### 1. Изоляция Данных Пользователей

**ID:** user-data-isolation.1

**User Story:** Как пользователь, я хочу чтобы мои данные (настройки, задачи, контакты) были изолированы от других пользователей, чтобы при входе другого пользователя мои данные оставались приватными и восстанавливались при повторном входе.

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth)

#### Критерии Приемки

1.1. Система ДОЛЖНА определять пользователя по email из Google OAuth профиля

1.2. Таблица `user_data` ДОЛЖНА содержать колонку `user_email TEXT` для привязки данных к пользователю

1.3. КОГДА пользователь сохраняет данные, ТО система ДОЛЖНА автоматически добавлять `user_email` текущего авторизованного пользователя

1.4. КОГДА пользователь загружает данные, ТО система ДОЛЖНА фильтровать данные по `user_email` текущего авторизованного пользователя

1.5. КОГДА пользователь A выходит из системы (logout), ТО его данные ДОЛЖНЫ оставаться в базе данных

1.6. КОГДА пользователь B авторизуется, ТО система ДОЛЖНА показывать только данные пользователя B (фильтрация по `user_email`)

1.7. КОГДА пользователь A снова авторизуется, ТО система ДОЛЖНА восстановить все его данные из базы (фильтрация по `user_email`)

1.8. Изоляция данных ДОЛЖНА применяться ко всем типам данных:
   - Настройки AI Agent (settings.1)
   - Задачи (Tasks)
   - Контакты (Contacts)
   - Календарь (Calendar)
   - Любые другие пользовательские данные

1.9. Таблица `user_data` ДОЛЖНА быть обновлена вручную разработчиком (пересоздание базы):
   - Добавить колонку `user_email TEXT`
   - Создать индекс `idx_user_email` на колонке `user_email`

1.10. DataManager ДОЛЖЕН автоматически получать `user_email` текущего авторизованного пользователя из UserProfileManager

1.11. DataManager ДОЛЖЕН автоматически добавлять `user_email` при сохранении данных (без явного параметра в методе `saveData`)

1.12. DataManager ДОЛЖЕН автоматически фильтровать по `user_email` при загрузке данных (без явного параметра в методе `loadData`)

1.13. КОГДА пользователь не авторизован (нет email), ТО методы DataManager `saveData`/`loadData`/`deleteData` ДОЛЖНЫ возвращать ошибку "No user logged in"

1.14. UserProfileManager ДОЛЖЕН кэшировать email текущего пользователя в памяти (свойство `currentUserEmail`)

1.15. КОГДА пользователь успешно авторизуется через Google OAuth, ТО UserProfileManager ДОЛЖЕН установить `currentUserEmail` из профиля

1.16. КОГДА access token обновляется (refresh), ТО UserProfileManager ДОЛЖЕН обновить `currentUserEmail` из обновленного профиля

1.17. КОГДА приложение запускается И пользователь авторизован (есть валидные токены), ТО UserProfileManager ДОЛЖЕН загрузить профиль из базы данных И установить `currentUserEmail`

1.18. КОГДА пользователь выходит (logout), ТО UserProfileManager ДОЛЖЕН очистить `currentUserEmail` (установить в `null`)

1.19. КОГДА DataManager возвращает ошибку "No user logged in" И пользователь не авторизован, ТО система ДОЛЖНА перенаправить на экран логина И очистить все кэши

1.20. КОГДА DataManager возвращает ошибку "No user logged in" И сессия истекла во время работы, ТО система ДОЛЖНА применить процедуру восстановления сессии (см. token-management-ui.1.1, token-management-ui.1.3) И при успешном восстановлении повторить операцию с данными без уведомления пользователю

1.21. КОГДА DataManager возвращает ошибку "No user logged in" во время logout (race condition), ТО система ДОЛЖНА молча игнорировать ошибку (логировать через "Logger" класс для отладки, см. clerkly.3)

1.22. Профиль пользователя (ключ `user_profile`) ДОЛЖЕН быть изолирован по email (каждый пользователь имеет свой профиль в базе данных)

1.23. Состояние окна (Window State) ДОЛЖНО быть изолировано по пользователям (каждый пользователь имеет свое положение окна)

1.24. OAuth токены ДОЛЖНЫ быть изолированы по пользователям (каждый пользователь имеет свои токены)

**Примечание:** Эта функциональность критична для multi-user support и должна быть реализована перед settings.1 (настройки AI Agent).

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Очистка данных пользователя при logout (данные сохраняются для восстановления при повторном входе)
- Миграция данных между пользователями
- Экспорт/импорт данных пользователя
- Удаление всех данных пользователя из системы
- Управление несколькими аккаунтами одновременно
