# Документ Требований: User Data Isolation

## Введение

Данный документ описывает требования к изоляции данных пользователей в приложении Clerkly, обеспечивая приватность и безопасность данных при использовании приложения несколькими пользователями.

## Глоссарий

- **User** - Пользователь приложения, идентифицируемый по email из Google OAuth
- **users Table** - Таблица пользователей с полями: user_id (TEXT PRIMARY KEY, 10 символов), name (TEXT), email (TEXT UNIQUE)
- **user_id** - Уникальный идентификатор пользователя (TEXT, случайная строка 10 символов), используемый для изоляции данных
- **Data Isolation** - Изоляция данных между пользователями по user_id
- **user_data Table** - Таблица в SQLite для хранения пользовательских данных
- **DataManager** - Класс для управления данными в локальной базе данных
- **UserProfileManager** - Класс для управления профилем пользователя и текущим user_id
- **currentUserId** - Кэшированный user_id текущего авторизованного пользователя
- **Race Condition** - Ситуация, когда операции выполняются одновременно и могут конфликтовать

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `.kiro/specs/clerkly/requirements.md`).

**Применение к User Data Isolation:**
- Все данные пользователей хранятся в базе данных с привязкой к `user_id`
- DataManager автоматически фильтрует данные по текущему пользователю
- При смене пользователя данные предыдущего пользователя остаются в базе и восстанавливаются при повторном входе
- UI автоматически обновляется при смене пользователя, отображая данные нового пользователя из базы

## Требования

### Требование 0: Таблица Пользователей

**ID:** user-data-isolation.0

**User Story:** Как система, я хочу хранить информацию о пользователях в отдельной таблице, чтобы использовать числовой идентификатор для изоляции данных вместо email.

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth)

#### Критерии Приемки

0.1. Система ДОЛЖНА иметь таблицу `users` со следующей структурой:
   - `user_id TEXT PRIMARY KEY` - уникальный идентификатор пользователя (случайная строка 10 символов)
   - `name TEXT` - имя пользователя из Google OAuth профиля
   - `email TEXT NOT NULL UNIQUE` - email пользователя из Google OAuth профиля

0.2. КОГДА пользователь впервые авторизуется через Google OAuth, ТО система ДОЛЖНА создать запись в таблице `users` с новым случайным user_id (10 символов, alphanumeric)

0.3. КОГДА пользователь повторно авторизуется, ТО система ДОЛЖНА найти существующую запись по email и использовать её user_id

0.4. Система ДОЛЖНА обновлять поле `name` при каждой авторизации (если имя изменилось в Google профиле)

0.5. Таблица `users` ДОЛЖНА иметь индекс на поле `email` для быстрого поиска

**Тестируемость:** Да - через модульные тесты создания/поиска пользователей, функциональные тесты авторизации

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should create user record on first login"
- `tests/functional/user-data-isolation.spec.ts` - "should find existing user on re-login"
- `tests/functional/user-data-isolation.spec.ts` - "should update user name if changed"

### Требование 1: Изоляция Данных Пользователей

**ID:** user-data-isolation.1

**User Story:** Как пользователь, я хочу чтобы мои данные (настройки, задачи, контакты) были изолированы от других пользователей, чтобы при входе другого пользователя мои данные оставались приватными и восстанавливались при повторном входе.

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth), user-data-isolation.0 (таблица пользователей)

#### Критерии Приемки

1.1. Система ДОЛЖНА определять пользователя по email из Google OAuth профиля и получать соответствующий user_id из таблицы `users`

1.2. Таблица `user_data` ДОЛЖНА содержать колонку `user_id TEXT` для привязки данных к пользователю

1.3. КОГДА пользователь сохраняет данные, ТО система ДОЛЖНА автоматически добавлять `user_id` текущего авторизованного пользователя

1.4. КОГДА пользователь загружает данные, ТО система ДОЛЖНА фильтровать данные по `user_id` текущего авторизованного пользователя

1.5. КОГДА пользователь A выходит из системы (logout), ТО его данные ДОЛЖНЫ оставаться в базе данных

1.6. КОГДА пользователь B авторизуется, ТО система ДОЛЖНА показывать только данные пользователя B (фильтрация по `user_id`)

1.7. КОГДА пользователь A снова авторизуется, ТО система ДОЛЖНА восстановить все его данные из базы (фильтрация по `user_id`)

1.8. Изоляция данных ДОЛЖНА применяться ко всем типам пользовательских данных:
   - Настройки AI Agent (settings.1)
   - Профиль пользователя (account-profile.1)
   - OAuth токены (google-oauth-auth.4)
   - Агенты (agents-list)
   - Любые другие пользовательские данные

1.9. Изоляция данных НЕ ДОЛЖНА применяться к глобальным настройкам приложения:
   - Состояние окна (Window State) - см. window-management.5.7
   - Другие глобальные настройки интерфейса

**Обоснование:** Состояние окна является глобальным для обеспечения консистентного поведения окна независимо от авторизованного пользователя.

1.10. Таблица `user_data` ДОЛЖНА быть обновлена:
   - Изменить колонку `user_email TEXT` на `user_id TEXT`
   - Создать индекс `idx_user_id` на колонке `user_id`

1.11. DataManager ДОЛЖЕН автоматически получать `user_id` текущего авторизованного пользователя из UserProfileManager

1.12. DataManager ДОЛЖЕН автоматически добавлять `user_id` при сохранении данных (без явного параметра в методе `saveData`)

1.13. DataManager ДОЛЖЕН автоматически фильтровать по `user_id` при загрузке данных (без явного параметра в методе `loadData`)

1.14. КОГДА пользователь не авторизован (нет user_id), ТО методы DataManager `saveData`/`loadData`/`deleteData` ДОЛЖНЫ возвращать ошибку "No user logged in"

1.15. UserProfileManager ДОЛЖЕН кэшировать user_id текущего пользователя в памяти (свойство `currentUserId`)

1.16. КОГДА пользователь успешно авторизуется через Google OAuth, ТО UserProfileManager ДОЛЖЕН:
   - Найти или создать запись в таблице `users` по email
   - Установить `currentUserId` из найденной/созданной записи

1.17. КОГДА приложение запускается И пользователь авторизован (есть валидные токены), ТО UserProfileManager ДОЛЖЕН загрузить профиль из базы данных И установить `currentUserId`

1.18. КОГДА пользователь выходит (logout), ТО UserProfileManager ДОЛЖЕН очистить `currentUserId` (установить в `null`)

1.19. КОГДА DataManager возвращает ошибку "No user logged in" И пользователь не авторизован, ТО система ДОЛЖНА перенаправить на экран логина И очистить все кэши

1.20. КОГДА DataManager возвращает ошибку "No user logged in" И сессия истекла во время работы, ТО система ДОЛЖНА применить процедуру восстановления сессии (см. token-management-ui.1.1, token-management-ui.1.3) И при успешном восстановлении повторить операцию с данными без уведомления пользователю

1.21. КОГДА DataManager возвращает ошибку "No user logged in" во время logout (race condition), ТО система ДОЛЖНА молча игнорировать ошибку (логировать через "Logger" класс для отладки, см. clerkly.3)

**Примечание:** Эта функциональность критична для multi-user support и должна быть реализована перед settings.1 (настройки AI Agent).

**Тестируемость:** Да - через модульные тесты DataManager и UserProfileManager, property-based тесты изоляции данных, функциональные тесты multi-user сценариев

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should isolate data between different users"
- `tests/functional/user-data-isolation.spec.ts` - "should restore user data after re-login"
- `tests/functional/user-data-isolation.spec.ts` - "should persist data after logout"
- `tests/functional/user-data-isolation.spec.ts` - "should filter data by user_id"

#### Нефункциональные Требования

**Производительность:**
- Фильтрация данных по user_id должна выполняться менее чем за 50ms
- Индекс на user_id должен обеспечивать быстрый поиск данных пользователя

**Безопасность:**
- Данные одного пользователя не должны быть доступны другому пользователю ни при каких условиях
- При отсутствии авторизации операции с данными должны возвращать ошибку (fail-safe)

**Надежность:**
- Данные пользователя должны сохраняться после logout для восстановления при повторном входе
- Race conditions во время logout не должны приводить к потере данных

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Очистка данных пользователя при logout (данные сохраняются для восстановления при повторном входе)
- Миграция данных между пользователями
- Экспорт/импорт данных пользователя
- Удаление всех данных пользователя из системы
- Управление несколькими аккаунтами одновременно
