# Документ Требований: User Data Isolation

## Введение

Данный документ описывает требования к изоляции данных пользователей в приложении Clerkly, обеспечивая приватность и безопасность данных при использовании приложения несколькими пользователями на одном устройстве.

## Глоссарий

- **User** - Пользователь приложения, идентифицируемый по email из Google OAuth
- **users Table** - Таблица пользователей с полями: user_id (TEXT PRIMARY KEY), name (TEXT), email (TEXT UNIQUE)
- **user_id** - Уникальный идентификатор пользователя (TEXT, случайная alphanumeric строка 10 символов)
- **user_data Table** - Таблица в SQLite для хранения пользовательских данных с привязкой к user_id
- **Data Isolation** - Механизм изоляции данных между пользователями по user_id
- **UserManager** - Класс для управления пользователями (создание, поиск, обновление)
- **DataManager** - Класс для управления данными в локальной базе данных с автоматической фильтрацией по user_id
- **UserProfileManager** - Класс для управления профилем пользователя и текущим user_id
- **currentUserId** - Кэшированный user_id текущего авторизованного пользователя в памяти

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `.kiro/specs/clerkly/requirements.md`).

**Применение к User Data Isolation:**
- Все данные пользователей хранятся в базе данных с привязкой к `user_id`
- DataManager автоматически фильтрует данные по текущему пользователю
- При смене пользователя данные предыдущего пользователя остаются в базе и восстанавливаются при повторном входе
- UI автоматически обновляется при смене пользователя, отображая данные нового пользователя из базы

## Требования

### Требование 0: Таблица Пользователей

**ID:** user-data-isolation.0

**User Story:** Как система, я хочу хранить информацию о пользователях в отдельной таблице, чтобы использовать стабильный идентификатор для изоляции данных вместо email (который может измениться).

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth)

#### Критерии Приемки

0.1. Система ДОЛЖНА иметь таблицу `users` со следующей структурой:
   - `user_id TEXT PRIMARY KEY` - уникальный идентификатор пользователя
   - `name TEXT` - имя пользователя из Google OAuth профиля (может быть NULL)
   - `email TEXT NOT NULL UNIQUE` - email пользователя из Google OAuth профиля

0.2. `user_id` ДОЛЖЕН генерироваться как случайная alphanumeric строка длиной 10 символов (A-Z, a-z, 0-9)

0.3. КОГДА пользователь впервые авторизуется через Google OAuth, ТО система ДОЛЖНА создать запись в таблице `users` с новым сгенерированным user_id

0.4. КОГДА пользователь повторно авторизуется, ТО система ДОЛЖНА найти существующую запись по email и использовать её user_id

0.5. Система ДОЛЖНА обновлять поле `name` при каждой авторизации, если имя изменилось в Google профиле

0.6. Таблица `users` ДОЛЖНА иметь индекс `idx_users_email` на поле `email` для быстрого поиска

**Тестируемость:** Да - через модульные тесты UserManager, функциональные тесты авторизации

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should create user record on first login"
- `tests/functional/user-data-isolation.spec.ts` - "should find existing user on re-login"
- `tests/functional/user-data-isolation.spec.ts` - "should update user name if changed"

### Требование 1: Класс UserManager

**ID:** user-data-isolation.1

**User Story:** Как разработчик, я хочу иметь централизованный класс для управления пользователями, чтобы логика создания и поиска пользователей была инкапсулирована в одном месте.

**Зависимости:** user-data-isolation.0 (таблица users)

#### Критерии Приемки

1.1. Система ДОЛЖНА иметь класс `UserManager` в файле `src/main/auth/UserManager.ts`

1.2. UserManager ДОЛЖЕН иметь приватный метод `generateUserId(): string`, который генерирует случайную alphanumeric строку длиной 10 символов

1.3. UserManager ДОЛЖЕН иметь публичный метод `findOrCreateUser(email: string, name: string | null): User`, который:
   - Ищет пользователя по email в таблице `users`
   - Если найден - возвращает существующего пользователя (обновляя name при необходимости)
   - Если не найден - создает нового пользователя с сгенерированным user_id

1.4. UserManager ДОЛЖЕН возвращать объект типа `User`:
   ```typescript
   interface User {
     user_id: string;
     name: string | null;
     email: string;
   }
   ```

1.5. UserManager ДОЛЖЕН использовать синхронные операции с базой данных (better-sqlite3)

**Тестируемость:** Да - через модульные тесты UserManager

#### Функциональные Тесты

- Покрываются тестами из user-data-isolation.0

### Требование 2: Изоляция Данных в user_data

**ID:** user-data-isolation.2

**User Story:** Как пользователь, я хочу чтобы мои данные были изолированы от других пользователей, чтобы при входе другого пользователя мои данные оставались приватными.

**Зависимости:** user-data-isolation.0 (таблица users), user-data-isolation.1 (UserManager)

#### Критерии Приемки

2.1. Таблица `user_data` ДОЛЖНА содержать колонку `user_id TEXT NOT NULL` для привязки данных к пользователю

2.2. Таблица `user_data` ДОЛЖНА иметь FOREIGN KEY на `users(user_id)`

2.3. Таблица `user_data` ДОЛЖНА иметь составной PRIMARY KEY `(key, user_id)`

2.4. Таблица `user_data` ДОЛЖНА иметь индекс `idx_user_id` на колонке `user_id`

2.5. КОГДА пользователь сохраняет данные через DataManager, ТО система ДОЛЖНА автоматически добавлять `user_id` текущего пользователя

2.6. КОГДА пользователь загружает данные через DataManager, ТО система ДОЛЖНА автоматически фильтровать по `user_id` текущего пользователя

2.7. КОГДА пользователь удаляет данные через DataManager, ТО система ДОЛЖНА удалять только данные с `user_id` текущего пользователя

2.8. Изоляция данных ДОЛЖНА применяться ко всем типам пользовательских данных:
   - Настройки AI Agent (settings.1)
   - Профиль пользователя (account-profile.1)
   - OAuth токены (google-oauth-auth.4)
   - Любые другие пользовательские данные

2.9. Изоляция данных НЕ ДОЛЖНА применяться к глобальным настройкам приложения:
   - Состояние окна (Window State) - см. window-management.5.7

**Обоснование:** Состояние окна является глобальным для обеспечения консистентного поведения окна независимо от авторизованного пользователя.

**Тестируемость:** Да - через модульные тесты DataManager, property-based тесты изоляции

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should isolate data between different users"
- `tests/functional/user-data-isolation.spec.ts` - "should filter data by user_id"

### Требование 3: Интеграция UserProfileManager с UserManager

**ID:** user-data-isolation.3

**User Story:** Как система, я хочу чтобы UserProfileManager управлял текущим user_id, чтобы DataManager мог автоматически фильтровать данные.

**Зависимости:** user-data-isolation.1 (UserManager), google-oauth-auth.3 (авторизация)

#### Критерии Приемки

3.1. UserProfileManager ДОЛЖЕН иметь зависимость на UserManager

3.2. UserProfileManager ДОЛЖЕН кэшировать `user_id` текущего пользователя в приватном свойстве `currentUserId: string | null`

3.3. UserProfileManager ДОЛЖЕН иметь публичный метод `getCurrentUserId(): string | null`

3.4. КОГДА пользователь успешно авторизуется через Google OAuth, ТО UserProfileManager ДОЛЖЕН:
   - Вызвать `userManager.findOrCreateUser(profile.email, profile.name)`
   - Установить `currentUserId` из возвращенного объекта User

3.5. КОГДА приложение запускается И пользователь авторизован (есть валидные токены), ТО UserProfileManager ДОЛЖЕН:
   - Загрузить профиль из базы данных
   - Вызвать `userManager.findOrCreateUser(profile.email, profile.name)`
   - Установить `currentUserId`

3.6. КОГДА пользователь выходит (logout), ТО UserProfileManager ДОЛЖЕН очистить `currentUserId` (установить в `null`)

**Тестируемость:** Да - через модульные тесты UserProfileManager

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should restore user data after re-login"

### Требование 4: Интеграция DataManager с UserProfileManager

**ID:** user-data-isolation.4

**User Story:** Как разработчик, я хочу чтобы DataManager автоматически использовал user_id без явной передачи в каждый метод.

**Зависимости:** user-data-isolation.3 (UserProfileManager с currentUserId)

#### Критерии Приемки

4.1. DataManager ДОЛЖЕН иметь зависимость на UserProfileManager

4.2. DataManager ДОЛЖЕН автоматически получать `user_id` через `userProfileManager.getCurrentUserId()` в методах `saveData`, `loadData`, `deleteData`

4.3. КОГДА `getCurrentUserId()` возвращает `null`, ТО методы DataManager ДОЛЖНЫ выбрасывать ошибку "No user logged in"

4.4. Методы DataManager НЕ ДОЛЖНЫ принимать `user_id` как параметр (автоматическая изоляция)

**Тестируемость:** Да - через модульные тесты DataManager

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should persist data after logout"

### Требование 5: Обработка Ошибок Изоляции

**ID:** user-data-isolation.5

**User Story:** Как система, я хочу корректно обрабатывать ошибки изоляции данных, чтобы обеспечить безопасность и надежность.

**Зависимости:** user-data-isolation.4 (DataManager с изоляцией)

#### Критерии Приемки

5.1. КОГДА DataManager возвращает ошибку "No user logged in" И пользователь не авторизован, ТО система ДОЛЖНА перенаправить на экран логина

5.2. КОГДА DataManager возвращает ошибку "No user logged in" И сессия истекла во время работы, ТО система ДОЛЖНА:
   - Применить процедуру восстановления сессии (см. token-management-ui.1.1, token-management-ui.1.3)
   - При успешном восстановлении повторить операцию без уведомления пользователю

5.3. КОГДА DataManager возвращает ошибку "No user logged in" во время logout (race condition), ТО система ДОЛЖНА:
   - Молча игнорировать ошибку
   - Логировать через Logger класс для отладки (см. clerkly.3)

5.4. Данные одного пользователя НЕ ДОЛЖНЫ быть доступны другому пользователю ни при каких условиях (fail-safe)

**Тестируемость:** Да - через модульные и функциональные тесты обработки ошибок

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should handle No user logged in error"
- `tests/functional/user-data-isolation.spec.ts` - "should retry operation after token refresh"

### Требование 6: Миграция Существующих Данных

**ID:** user-data-isolation.6

**User Story:** Как система, я хочу мигрировать существующие данные с user_email на user_id, чтобы обеспечить обратную совместимость.

**Зависимости:** user-data-isolation.0 (таблица users), user-data-isolation.2 (новая схема user_data)

#### Критерии Приемки

6.1. Миграция ДОЛЖНА создать записи в таблице `users` для всех уникальных email из существующей таблицы `user_data`

6.2. Миграция ДОЛЖНА обновить колонку `user_email` на `user_id` в таблице `user_data`

6.3. Миграция ДОЛЖНА связать существующие данные с соответствующими user_id

6.4. Миграция ДОЛЖНА быть идемпотентной (повторный запуск не должен создавать дубликаты)

6.5. Миграция ДОЛЖНА иметь DOWN секцию для отката изменений

**Тестируемость:** Да - через модульные тесты миграции

#### Функциональные Тесты

- Покрываются тестами из user-data-isolation.2

## Нефункциональные Требования

### Производительность

- Фильтрация данных по user_id ДОЛЖНА выполняться менее чем за 50ms
- Индекс на user_id ДОЛЖЕН обеспечивать O(log n) поиск данных пользователя
- Генерация user_id ДОЛЖНА выполняться менее чем за 1ms

### Безопасность

- Данные одного пользователя НЕ ДОЛЖНЫ быть доступны другому пользователю ни при каких условиях
- При отсутствии авторизации операции с данными ДОЛЖНЫ возвращать ошибку (fail-safe)
- user_id НЕ ДОЛЖЕН содержать персональную информацию (случайная строка)

### Надежность

- Данные пользователя ДОЛЖНЫ сохраняться после logout для восстановления при повторном входе
- Race conditions во время logout НЕ ДОЛЖНЫ приводить к потере данных
- Миграция данных ДОЛЖНА быть атомарной (транзакция)

---

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Очистка данных пользователя при logout (данные сохраняются для восстановления)
- Миграция данных между пользователями
- Экспорт/импорт данных пользователя
- Удаление всех данных пользователя из системы
- Управление несколькими аккаунтами одновременно
- Шифрование данных пользователя
