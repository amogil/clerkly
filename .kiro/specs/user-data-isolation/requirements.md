# Документ Требований: User Data Isolation

## Введение

Данный документ описывает требования к изоляции данных пользователей в приложении Clerkly, обеспечивая приватность и безопасность данных при использовании приложения несколькими пользователями на одном устройстве.

## Глоссарий

- **User** - Пользователь приложения, идентифицируемый по email из Google OAuth
- **users Table** - Таблица пользователей с полями: user_id (TEXT PRIMARY KEY), name (TEXT), email (TEXT UNIQUE)
- **user_id** - Уникальный идентификатор пользователя (TEXT, случайная alphanumeric строка 10 символов)
- **user_data Table** - Таблица в SQLite для хранения пользовательских настроек (key-value) с привязкой к user_id
- **Data Isolation** - Механизм изоляции данных между пользователями по user_id
- **DatabaseManager** - Единая точка входа для доступа к базе данных, инициализации и получения текущего user_id. Предоставляет репозитории для работы с данными.
- **Repository Pattern** - Паттерн доступа к данным через типизированные репозитории (SettingsRepository, UsersRepository и др.)
- **UserSettingsManager** - Класс для управления пользовательскими настройками (key-value в таблице user_data) с автоматической фильтрацией по user_id. Использует `dbManager.settings` репозиторий.
- **UserProfileManager** - Класс для управления профилем пользователя, таблицей users и текущим user_id
- **currentUserId** - Кэшированный user_id текущего авторизованного пользователя в памяти

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `.kiro/specs/clerkly/requirements.md`).

**Применение к User Data Isolation:**
- Все данные пользователей хранятся в базе данных с привязкой к `user_id`
- DatabaseManager предоставляет единую точку доступа к БД и текущему user_id
- UserSettingsManager и другие менеджеры используют DatabaseManager для автоматической фильтрации по user_id
- При смене пользователя данные предыдущего пользователя остаются в базе и восстанавливаются при повторном входе
- UI автоматически обновляется при смене пользователя, отображая данные нового пользователя из базы

## Требования

### Требование 0: Таблица Пользователей

**ID:** user-data-isolation.0

**User Story:** Как система, я хочу хранить информацию о пользователях в отдельной таблице, чтобы использовать стабильный идентификатор для изоляции данных вместо email.

**Зависимости:** google-oauth-auth.3 (авторизация через Google OAuth)

#### Критерии Приемки

0.1. Система ДОЛЖНА иметь таблицу `users` со следующей структурой:
   - `user_id TEXT PRIMARY KEY` - уникальный идентификатор пользователя
   - `name TEXT` - имя пользователя из Google OAuth профиля (может быть NULL)
   - `email TEXT NOT NULL UNIQUE` - email пользователя из Google OAuth профиля

0.2. `user_id` ДОЛЖЕН генерироваться как случайная alphanumeric строка длиной 10 символов (A-Z, a-z, 0-9) согласно алгоритму генерации

0.3. КОГДА пользователь авторизуется через Google OAuth, система ДОЛЖНА:
   - Найти запись в таблице `users` по email
   - Если запись существует - использовать её user_id
   - Если запись не существует - создать новую с сгенерированным user_id

0.4. Система ДОЛЖНА обновлять поле `name` при авторизации, если имя изменилось в Google профиле

0.5. Таблица `users` ДОЛЖНА иметь индекс `idx_users_email` на поле `email` для быстрого поиска

**Тестируемость:** Да - через модульные тесты UserProfileManager, функциональные тесты авторизации

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should create user record on first login"
- `tests/functional/user-data-isolation.spec.ts` - "should find existing user on re-login"
- `tests/functional/user-data-isolation.spec.ts` - "should update user name if changed"

### Требование 1: Управление Идентификацией Пользователя

**ID:** user-data-isolation.1

**User Story:** Как система, я хочу централизованно управлять идентификацией пользователя с персистентным хранением, чтобы обеспечить корректную изоляцию данных и восстановление сессии между перезапусками.

**Зависимости:** user-data-isolation.0 (таблица users)

#### Критерии Приемки

1.1. Система ДОЛЖНА определять текущего пользователя по его `user_id`

1.2. КОГДА пользователь успешно авторизуется через Google OAuth, система ДОЛЖНА:
   - Получить токены от Google
   - Загрузить профиль пользователя (используя полученный access token)
   - Найти или создать запись пользователя по email
   - Сохранить `user_id` в глобальное хранилище (таблица `user_data` с ключом `current_user_id` и `userId = '__system__'`)
   - Сохранить токены (теперь `user_id` установлен)
   - Показать экран Agents

1.3. КОГДА приложение запускается, система ДОЛЖНА:
   - Загрузить сохраненный `user_id` из глобального хранилища
   - Если `user_id` НЕ найден → показать экран Login
   - Если `user_id` найден:
     - Загрузить пользователя из таблицы `users` по `user_id`
     - Если пользователь НЕ найден → очистить сохраненный `user_id`, показать Login
     - Если пользователь найден → показать экран Agents

1.4. КОГДА пользователь выходит (logout), система ДОЛЖНА:
   - Попытаться отозвать токены в Google (если ошибка - логировать, но НЕ прерывать процесс выхода)
   - Удалить токены из БД
   - Удалить `user_id` из глобального хранилища
   - Очистить информацию о текущем пользователе из памяти
   - Показать экран Login
   - НЕ удалять данные пользователя (агенты, сообщения, настройки остаются в БД)

1.5. Система ДОЛЖНА предоставлять способ получить `user_id` текущего пользователя для изоляции данных через кэширование:
   - Первый вызов `getCurrentUserId()` читает из БД
   - Последующие вызовы возвращают кэшированное значение
   - Кэш сбрасывается при `setCurrentUser()` и `clearSession()`

1.6. Глобальное хранилище `user_id` ДОЛЖНО использовать тот же паттерн, что и window state:
   - Ключ: `current_user_id`
   - userId: `__system__` (системный пользователь)
   - Хранится в таблице `user_data`

**Тестируемость:** Да - через модульные тесты UserManager, GlobalRepository

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should restore user data after re-login"
- `tests/functional/user-data-isolation.spec.ts` - "should persist userId between app restarts"
- `tests/functional/user-data-isolation.spec.ts` - "should clear userId on logout"

### Требование 2: Изоляция Данных в user_data

**ID:** user-data-isolation.2

**User Story:** Как пользователь, я хочу чтобы мои данные были изолированы от других пользователей, чтобы при входе другого пользователя мои данные оставались приватными.

**Зависимости:** user-data-isolation.1 (UserProfileManager с currentUserId)

#### Критерии Приемки

2.1. Таблица `user_data` ДОЛЖНА содержать колонку `user_id TEXT NOT NULL` для привязки данных к пользователю

2.2. Таблица `user_data` ДОЛЖНА иметь составной PRIMARY KEY `(key, user_id)`

2.3. Таблица `user_data` ДОЛЖНА иметь индекс `idx_user_id` на колонке `user_id`

2.4. КОГДА пользователь сохраняет данные через UserSettingsManager, система ДОЛЖНА автоматически добавлять `user_id` текущего пользователя

2.5. КОГДА пользователь загружает данные через UserSettingsManager, система ДОЛЖНА автоматически фильтровать по `user_id` текущего пользователя

2.6. КОГДА пользователь удаляет данные через UserSettingsManager, система ДОЛЖНА удалять только данные с `user_id` текущего пользователя

2.7. Изоляция данных ДОЛЖНА применяться ко всем типам пользовательских данных:
   - Настройки AI Agent (settings.1)
   - Профиль пользователя (account-profile.1)
   - OAuth токены (google-oauth-auth.4)
   - Любые другие пользовательские данные

2.8. Изоляция данных НЕ ДОЛЖНА применяться к глобальным настройкам приложения:
   - Состояние окна (Window State) - см. window-management.5.7

2.9. Данные с user_id, который не существует в таблице users, ДОЛЖНЫ оставаться в базе данных (не удаляться автоматически). Они просто не будут доступны ни одному пользователю.

**Обоснование:** Состояние окна является глобальным для обеспечения консистентного поведения окна независимо от авторизованного пользователя. Целостность данных поддерживается логикой приложения, а не FOREIGN KEY constraints.

**Тестируемость:** Да - через модульные тесты UserSettingsManager, property-based тесты изоляции

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should isolate data between different users"
- `tests/functional/user-data-isolation.spec.ts` - "should filter data by user_id"

### Требование 3: Автоматическая Изоляция в UserSettingsManager

**ID:** user-data-isolation.3

**User Story:** Как разработчик, я хочу чтобы изоляция данных происходила автоматически, без необходимости явно указывать пользователя в каждом вызове.

3.3. API UserSettingsManager НЕ ДОЛЖЕН требовать явной передачи `user_id` (автоматическая изоляция)

### Требование 4: Обработка Ошибок Изоляции

**ID:** user-data-isolation.4

**User Story:** Как система, я хочу корректно обрабатывать ошибки изоляции данных, чтобы обеспечить безопасность и надежность.

**Зависимости:** user-data-isolation.3 (UserSettingsManager с изоляцией), user-data-isolation.6 (DatabaseManager)

#### Критерии Приемки

4.1. КОГДА DatabaseManager возвращает ошибку "No user logged in" И пользователь не авторизован, система ДОЛЖНА перенаправить на экран логина

4.2. КОГДА DatabaseManager возвращает ошибку "No user logged in" И сессия истекла во время работы, система ДОЛЖНА:
   - Применить процедуру восстановления сессии (см. token-management-ui.1.1, token-management-ui.1.3)
   - При успешном восстановлении повторить операцию без уведомления пользователю

4.3. КОГДА DatabaseManager возвращает ошибку "No user logged in" во время logout (race condition), система ДОЛЖНА:
   - Молча игнорировать ошибку
   - Логировать через Logger класс для отладки (см. clerkly.3)

4.4. Данные одного пользователя НЕ ДОЛЖНЫ быть доступны другому пользователю ни при каких условиях (fail-safe)

**Тестируемость:** Да - через модульные и функциональные тесты обработки ошибок

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should handle No user logged in error"
- `tests/functional/user-data-isolation.spec.ts` - "should retry operation after token refresh"

### Требование 6: Единая Точка Доступа к Базе Данных

**ID:** user-data-isolation.6

**User Story:** Как разработчик, я хочу иметь единую точку доступа к базе данных с автоматической изоляцией по user_id, чтобы все менеджеры данных работали консистентно.

**Зависимости:** user-data-isolation.1 (UserProfileManager с currentUserId)

#### Критерии Приемки

6.1. Система ДОЛЖНА иметь класс DatabaseManager как единую точку входа для:
   - Инициализации базы данных и миграций
   - Предоставления типизированных репозиториев для работы с данными
   - Автоматической подстановки user_id в запросы через репозитории

6.2. ВСЕ операции с данными ДОЛЖНЫ выполняться через репозитории DatabaseManager:
   - `dbManager.settings` — для пользовательских настроек (key-value)
   - `dbManager.agents` — для агентов
   - `dbManager.messages` — для сообщений
   - `dbManager.users` — для пользователей
   - `dbManager.global` — для глобальных данных (window state)

6.3. Репозитории ДОЛЖНЫ автоматически добавлять user_id текущего пользователя при операциях с пользовательскими данными

6.4. КОГДА текущий пользователь не определён, репозитории ДОЛЖНЫ выбрасывать ошибку "No user logged in"

6.5. Все менеджеры данных (UserSettingsManager, AgentManager, MessageManager и др.) ДОЛЖНЫ использовать репозитории DatabaseManager для выполнения операций

6.6. Менеджеры данных НЕ ДОЛЖНЫ требовать явной передачи user_id в публичных методах — user_id подставляется автоматически через репозитории

6.7. DatabaseManager ДОЛЖЕН инициализироваться при старте приложения до создания других менеджеров

6.8. WindowStateManager является ИСКЛЮЧЕНИЕМ — он работает с глобальными данными и использует `dbManager.global.windowState`

6.9. DatabaseManager МОЖЕТ предоставлять метод `getDatabase()` для низкоуровневого доступа, но его использование ДОЛЖНО быть ограничено:
   - Миграциями
   - Тестами

**Тестируемость:** Да - через модульные тесты DatabaseManager

#### Функциональные Тесты

- `tests/functional/user-data-isolation.spec.ts` - "should provide database access through DatabaseManager"
- `tests/functional/user-data-isolation.spec.ts` - "should provide current userId through DatabaseManager"

### Требование 7: Drizzle ORM и Repository Pattern

**ID:** user-data-isolation.7

**User Story:** Как разработчик, я хочу использовать type-safe ORM с Repository Pattern, чтобы гарантировать изоляцию данных и получить автодополнение в IDE.

**Зависимости:** user-data-isolation.6 (DatabaseManager)

#### Критерии Приемки

7.1. Система ДОЛЖНА использовать Drizzle ORM для работы с базой данных:
   - `drizzle-orm` — основной ORM
   - `drizzle-kit` — CLI для миграций (devDependency)

7.2. Drizzle ДОЛЖЕН быть настроен для работы с better-sqlite3 (синхронный режим)

7.3. Схема таблиц ДОЛЖНА находиться в `src/main/db/schema.ts` и описывать:
   - Таблицу `users` (user_id, name, email, google_id, locale, last_synced)
   - Таблицу `user_data` (key, user_id, value, created_at, updated_at)
   - Таблицу `agents` (agent_id, user_id, name, created_at, updated_at, archived_at)
   - Таблицу `messages` (id, agent_id, timestamp, payload_json)

7.4. Все таблицы ДОЛЖНЫ экспортировать TypeScript типы для использования в коде

7.5. Drizzle instance НЕ ДОЛЖЕН быть доступен снаружи DatabaseManager (приватный)

7.6. Каждый репозиторий ДОЛЖЕН предоставлять типизированные методы для CRUD операций

7.7. Репозиторий `users` ДОЛЖЕН работать без текущего userId (для авторизации):
   - `users.findByEmail(email)` — поиск пользователя по email
   - `users.findOrCreate(email, name)` — создание или поиск пользователя
   - `users.update(userId, data)` — обновление данных пользователя

7.8. Репозиторий `global` ДОЛЖЕН предоставлять доступ к данным без фильтрации по userId:
   - `global.windowState` — состояние окна

**Тестируемость:** Да - через модульные тесты репозиториев

## Нефункциональные Требования

### Производительность

- Фильтрация данных по пользователю ДОЛЖНА выполняться менее чем за 50ms
- Поиск данных пользователя ДОЛЖЕН быть эффективным (O(log n))
- Генерация идентификатора пользователя ДОЛЖНА выполняться менее чем за 1ms

### Безопасность

- Данные одного пользователя НЕ ДОЛЖНЫ быть доступны другому пользователю ни при каких условиях
- При отсутствии авторизации операции с данными ДОЛЖНЫ возвращать ошибку (fail-safe)
- Идентификатор пользователя НЕ ДОЛЖЕН содержать персональную информацию

### Надежность

- Данные пользователя ДОЛЖНЫ сохраняться после logout для восстановления при повторном входе
- Race conditions во время logout НЕ ДОЛЖНЫ приводить к потере данных
- Обновление схемы данных ДОЛЖНО быть атомарным

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Очистка данных пользователя при logout (данные сохраняются для восстановления)
- Миграция данных между пользователями
