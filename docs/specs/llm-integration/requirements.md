# Документ Требований: LLM Integration

## Введение

Данный документ описывает требования к интеграции LLM в приложение Clerkly. Функциональность обеспечивает полный цикл взаимодействия пользователя с AI-агентом: отправка сообщения → вызов LLM → стриминг reasoning → получение ответа → отображение в UI.

## Глоссарий

- **MainPipeline** — оркестратор LLM-цикла в main process
- **PromptBuilder** — компонент сборки промпта (системный промпт + YAML-история + инструменты)
- **LLMProvider** — провайдер LLM (OpenAI, Anthropic, Google)
- **Reasoning** — внутренние "размышления" модели перед ответом (стримится в реальном времени)
- **Action** — структурированный ответ модели (structured output, приходит целиком)
- **kind** — тип сообщения: `user | llm | error`
- **reply_to_message_id** — ссылка на сообщение, на которое даётся ответ (колонка `messages.reply_to_message_id`, null только у первого)

## Требования

### 1. LLM-цикл (MainPipeline)

**ID:** llm-integration.1

**User Story:** Как пользователь, я хочу чтобы после отправки сообщения агент автоматически обращался к LLM и показывал ответ в чате.

#### Критерии Приемки

1.1. КОГДА пользователь отправляет сообщение (`kind: user`), ТО `MainPipeline` ДОЛЖЕН запускаться асинхронно в фоне — IPC возвращает результат немедленно

1.2. `MainPipeline` ДОЛЖЕН быть stateless — всё состояние выполнения хранится в локальных переменных `run()`, что обеспечивает параллельное выполнение для разных агентов

1.3. `MainPipeline` ДОЛЖЕН получать API ключ из `UserSettingsManager` через `loadAPIKey(provider)`

1.4. `MainPipeline` ДОЛЖЕН использовать `PromptBuilder` для сборки промпта перед вызовом LLM

1.5. `MainPipeline` ДОЛЖЕН создавать `kind: llm` сообщение при получении первого reasoning чанка или при завершении генерации

1.6. `MainPipeline` ДОЛЖЕН обновлять `kind: llm` сообщение при каждом reasoning чанке и при получении финального action

1.7. `MainPipeline` ДОЛЖЕН проставлять `messages.reply_to_message_id` для каждого создаваемого сообщения:
  - Первое сообщение в чате агента — `null`
  - Все остальные сообщения — `id` предыдущего сообщения

---

### 2. Стриминг и события

**ID:** llm-integration.2

**User Story:** Как пользователь, я хочу видеть reasoning модели в реальном времени, пока она "думает".

#### Критерии Приемки

2.1. На каждый reasoning чанк ДОЛЖНЫ эмититься два события одновременно:
  - `message.llm.reasoning.updated` — с полями `{ messageId, agentId, delta, accumulatedText }`
  - `message.updated` — с полным snapshot сообщения

2.2. `message.created` ДОЛЖЕН эмититься при создании любого нового сообщения (user, llm, error)

2.3. `message.updated` ДОЛЖЕН эмититься при любом обновлении сообщения (промежуточном и финальном)

2.4. Renderer ДОЛЖЕН подписываться на `message.llm.reasoning.updated` для специфичной анимации reasoning

2.5. Renderer ДОЛЖЕН подписываться на `message.updated` для общего обновления состояния сообщения

---

### 3. Обработка ошибок

**ID:** llm-integration.3

**User Story:** Как пользователь, я хочу видеть понятное сообщение об ошибке если LLM недоступен или вернул ошибку.

#### Критерии Приемки

3.1. ЕСЛИ ошибка произошла до получения первого чанка (сеть, неверный ключ, таймаут), ТО ДОЛЖНО создаться только `kind: error` сообщение

3.2. ЕСЛИ ошибка произошла после начала стриминга, ТО:
  - Существующий `kind: llm` ДОЛЖЕН быть обновлён с флагом `"interrupted": true`
  - ДОЛЖНО создаться `kind: error` сообщение следом

3.3. `kind: error` сообщение ДОЛЖНО содержать текст ошибки в `data.error.message` и тип ошибки в `data.error.type`

3.4. Сообщения об ошибке в чате ДОЛЖНЫ отображаться как стандартизированный диалог ошибки, визуально отличимый от информационных и подтверждающих диалогов

3.4.1. Диалог ошибки ДОЛЖЕН иметь единый формат (заголовок, текст, опциональное действие), при этом текст сообщения зависит от причины ошибки

3.4.2. Диалог ошибки для LLM ДОЛЖЕН использовать один и тот же тип представления, но с разными пользовательскими сообщениями для:
  - отсутствия API ключа
  - ошибок авторизации (HTTP 401/403)

3.4.3. ЕСЛИ ошибка относится к отсутствию API ключа ИЛИ к ошибке авторизации (HTTP 401/403), ТО диалог ДОЛЖЕН содержать действия "Open Settings" (навигация к настройкам LLM Provider) и "Retry" (повтор последнего запроса и показ результата на основе последнего `kind:user` сообщения в БД)

3.4.4. Диалоги ошибок и уведомлений (включая rate limit) ДОЛЖНЫ занимать всю ширину области чата
3.4.5. КОГДА диалог содержит действия "Open Settings" и "Retry", ТО "Open Settings" ДОЛЖЕН быть основным (primary) действием, а "Retry" — вторичным
3.4.6. КОГДА пользователь нажимает "Retry" в диалоге ошибки авторизации/ключа, ТО сообщение об ошибке ДОЛЖНО скрываться и запрос ДОЛЖЕН повторяться

3.5. Типы ошибок и их сообщения:
  - **Нет API ключа**: `"API key is not set. Add it in Settings to continue."` + действие "Open Settings"
  - **Неверный ключ** (HTTP 401/403): `"Invalid API key. Please check your key and try again."` + действие "Open Settings"
  - **Нет сети / таймаут**: `"Network error. Please check your internet connection."`
  - **Rate limit** (HTTP 429): `"Rate limit exceeded. Please try again later."` — см. требование 3.7
  - **Внутренняя ошибка провайдера** (HTTP 5xx): `"Provider service unavailable. Please try again later."`
  - **Таймаут ожидания ответа**: `"Request timed out. The model took too long to respond."`

3.6. Таймаут ожидания ответа от LLM ДОЛЖЕН составлять 60 секунд. ЕСЛИ за это время не получен ни один чанк, ТО запрос прерывается с ошибкой таймаута

3.7. **Rate limit (HTTP 429) — стандартизированный диалог с обратным отсчётом:**

  3.7.1. ВМЕСТО обычного `kind: error` диалога при 429 ДОЛЖЕН отображаться диалог с обратным отсчётом (countdown) до автоматического повтора запроса

  3.7.2. Диалог ДОЛЖЕН содержать:
    - Текст: `"Rate limit exceeded. Retrying in N seconds..."`
    - Кнопку "Cancel" для отмены повтора

3.7.3. КОГДА отсчёт достигает нуля, ТО `MainPipeline` ДОЛЖЕН автоматически повторить последний запрос на основе последнего `kind:user` сообщения в БД

  3.7.4. ЕСЛИ пользователь нажимает "Cancel" во время отсчёта, ТО:
    - Повтор отменяется
    - Исходное `kind: user` сообщение ДОЛЖНО быть удалено из чата и БД
    - Баннер исчезает

  3.7.5. КОГДА повторный запрос завершается успешно, ТО баннер исчезает и в чате появляется обычный `kind: llm` ответ

  3.7.6. Время ожидания перед повтором определяется следующим образом (в порядке приоритета):
    1. Заголовок `retry-after` в HTTP-ответе (число секунд) — поддерживается Anthropic
    2. Парсинг времени из текста ошибки в теле ответа (например, `"Please try again in 10.384s"`) — OpenAI включает это в message
    3. Дефолт: 10 секунд — если ни заголовок, ни текст не содержат времени

3.8. `kind: error` сообщение ДОЛЖНО исчезать из UI КОГДА пользователь отправляет следующее сообщение в том же чате:
  - При создании нового `kind: user` сообщения все `kind: error` сообщения этого агента ДОЛЖНЫ быть помечены флагом `dismissed: true` в payload
  - Сообщения с `dismissed: true` НЕ ДОЛЖНЫ отображаться в UI — ни сразу, ни после перезагрузки приложения
  - Сообщения с `dismissed: true` остаются в БД (не удаляются) — для отладки и аудита
  - Это поведение симметрично `interrupted: true` у `kind: llm`

3.9. `kind: error` сообщения НЕ ДОЛЖНЫ включаться в историю при сборке промпта для LLM:
  - `PromptBuilder` ДОЛЖЕН фильтровать сообщения с `kind: error` при сериализации истории в YAML
  - Это гарантирует что LLM не видит технические ошибки как часть диалога

### 4. Построение промпта (PromptBuilder)

**ID:** llm-integration.4

**User Story:** Как разработчик, я хочу расширяемую систему сборки промпта для добавления новых фич без изменения архитектуры.

#### Критерии Приемки

4.1. `PromptBuilder` ДОЛЖЕН принимать системный промпт, список `AgentFeature` и `HistoryStrategy`

4.2. История сообщений ДОЛЖНА сериализоваться в YAML-формат для передачи LLM

4.3. `reasoning.text` ДОЛЖЕН исключаться из YAML-истории (`excluded_from_replay: true`)

4.4. Поле `model` ДОЛЖНО исключаться из YAML-истории

4.5. В MVP системный промпт — заглушка `"You are a helpful AI assistant."`

4.6. В MVP список фич пустой, история — полная (`FullHistoryStrategy`)

---

### 5. LLM Provider Interface

**ID:** llm-integration.5

**User Story:** Как разработчик, я хочу единый интерфейс для всех LLM провайдеров.

#### Критерии Приемки

5.1. `ILLMProvider` ДОЛЖЕН иметь метод `chat(messages, options, onChunk): Promise<LLMAction>`

5.2. `apiKey` ДОЛЖЕН передаваться в конструктор провайдера (не в каждый вызов)

5.2.1. `MainPipeline` ДОЛЖЕН создавать новый экземпляр провайдера при каждом вызове `run()`, читая актуальные настройки из `UserSettingsManager` — это гарантирует, что изменение провайдера или ключа в настройках вступает в силу при следующем запросе

5.3. Reasoning ДОЛЖЕН стримиться через `onChunk` с полями `{ type: 'reasoning', delta, done }`

5.4. Action ДОЛЖЕН возвращаться целиком как `LLMAction` после завершения генерации

5.5. В MVP `LLMAction.type` — только `'text'`

5.6. `LLMAction` ДОЛЖЕН содержать `usage` с полями: `input_tokens`, `output_tokens`, `total_tokens`, `cached_tokens?`, `reasoning_tokens?`

5.7. `OpenAIProvider` ДОЛЖЕН реализовывать structured output через `response_format: { type: "json_schema" }`

5.8. Для тестов используется модель `gpt-5-nano` с `reasoning_effort: "low"`, для прода — `gpt-5.2` с `reasoning_effort: "medium"`

### 6. Схема БД — колонка `kind`

**ID:** llm-integration.6

**User Story:** Как разработчик, я хочу хранить тип сообщения в отдельной колонке для эффективных запросов.

#### Критерии Приемки

6.1. Таблица `messages` ДОЛЖНА иметь колонку `kind TEXT NOT NULL` (без дефолта)

6.2. `kind` ДОЛЖЕН быть убран из `payload_json` — всегда передаётся явно при вставке

6.3. Миграция ДОЛЖНА проставить `kind = 'user'` всем существующим записям (все текущие сообщения в БД являются пользовательскими), после чего убрать поле `kind` из `payload_json`

6.4. `MessagesRepository.create()` ДОЛЖЕН принимать `kind` как обязательный параметр

---

### 7. Отображение в UI

**ID:** llm-integration.7

**User Story:** Как пользователь, я хочу видеть ответ агента в чате с reasoning и текстом ответа.

#### Критерии Приемки

7.1. `kind: llm` сообщение ДОЛЖНО отображаться как пузырь слева

7.2. Reasoning ДОЛЖЕН отображаться первым и стримиться в реальном времени

7.3. Текст ответа (action.content) ДОЛЖЕН появляться после завершения reasoning

7.4. `kind: error` ДОЛЖЕН отображаться как стандартизированный диалог ошибки в чате с текстом ошибки

7.5. Пока идёт стриминг reasoning, агент ДОЛЖЕН иметь статус `in-progress`

---

### 8. Прерывание текущего запроса при новом сообщении

**ID:** llm-integration.8

**User Story:** Как пользователь, я хочу отправить новое сообщение агенту пока он ещё отвечает, и получить ответ на оба сообщения сразу.

#### Критерии Приемки

8.1. КОГДА пользователь отправляет новое сообщение агенту, у которого есть активный `MainPipeline.run()`, ТО текущий запрос к LLM ДОЛЖЕН быть отменён через `AbortController`

8.2. Отмена выполняется только для текущего агента — запросы других агентов не затрагиваются

8.3. ЕСЛИ на момент отмены `kind: llm` сообщение ещё не было создано (агент не успел ответить ни одним чанком), ТО:
  - Запрос отменяется без создания каких-либо сообщений
  - Новое `kind: user` сообщение сохраняется в БД
  - Запускается новый `MainPipeline.run()` с новым сообщением

8.4. ЕСЛИ на момент отмены `kind: llm` сообщение уже было создано (агент начал отвечать), ТО:
  - Запрос отменяется
  - Существующий `kind: llm` помечается `interrupted: true` в payload
  - Новое `kind: user` сообщение сохраняется в БД
  - Запускается новый `MainPipeline.run()` с новым сообщением

8.5. Сообщения с `interrupted: true` НЕ ДОЛЖНЫ отображаться в UI (скрываются как будто их не было)

8.6. Сообщения с `interrupted: true` НЕ ДОЛЖНЫ включаться в историю при сборке промпта для LLM

8.7. Отменённый запрос НЕ ДОЛЖЕН создавать `kind: error` сообщение — это штатная ситуация, не ошибка

8.8. КОГДА агент архивируется, ТО активный `MainPipeline.run()` для этого агента (если есть) ДОЛЖЕН быть отменён через `AbortController` — без создания новых сообщений
