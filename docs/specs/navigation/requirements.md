# Документ Требований: Navigation

## Введение

Данный документ описывает требования к системе навигации и авторизации в приложении Clerkly, включая управление маршрутами, защищенные экраны и автоматическое перенаправление в зависимости от статуса авторизации пользователя.

## Глоссарий

- **Protected Routes** - Защищенные маршруты, доступные только авторизованным пользователям
- **Auth Guard** - Компонент, проверяющий авторизацию перед доступом к защищенным маршрутам
- **Login Screen** - Экран авторизации через Google OAuth
- **Agents** - Главный экран приложения после авторизации
- **Loader** - Индикатор загрузки, отображаемый во время асинхронных операций
- **Authorization Code** - Код авторизации, полученный от Google OAuth после успешной авторизации пользователя
- **LoginError Component** - Компонент для отображения ошибок авторизации
- **Access Token** - Токен доступа OAuth, используемый для авторизации API запросов
- **Refresh Token** - Токен обновления OAuth, используемый для получения нового access token

## Архитектурный Принцип

Приложение следует архитектурному принципу **Единого Источника Истины (Single Source of Truth)**, описанному в спецификации `clerkly` (см. `docs/specs/clerkly/requirements.md`).

**Примечание:** Управление токенами OAuth и обработка ошибок авторизации реализованы в спецификации google-oauth-auth. Данная спецификация описывает только UI-аспекты навигации.

## Требования

### 1. Навигация и Авторизация

**ID:** navigation.1

**User Story:** Как пользователь, я хочу чтобы приложение автоматически направляло меня на нужный экран в зависимости от статуса авторизации, чтобы не видеть функции, которые мне недоступны.

**Приоритет:** Высокий

**Зависимости:** google-oauth-auth.3, google-oauth-auth.11, google-oauth-auth.15

#### Критерии Приемки

1.1. КОГДА пользователь не авторизован, ТО приложение ДОЛЖНО показывать экран логина

1.2. КОГДА пользователь не авторизован, ТО пользователь НЕ МОЖЕТ получить доступ к защищенным экранам (Agents, Settings и другим защищенным маршрутам)

1.3. КОГДА пользователь нажимает кнопку "Continue with Google", ТО приложение ДОЛЖНО открыть системный браузер для авторизации

1.4. КОГДА браузер открыт, ТО кнопка "Continue with Google" ДОЛЖНА оставаться активной (пользователь может открыть несколько вкладок браузера)

1.5. КОГДА пользователь завершает авторизацию в браузере И authorization code получен, ТО приложение ДОЛЖНО показать loader с текстом "Signing in..."

1.6. КОГДА loader отображается, ТО приложение ДОЛЖНО выполнить следующие операции:
   - Обменять authorization code на токены
   - Загрузить профиль пользователя из Google UserInfo API (синхронно)

1.7. КОГДА пользователь успешно авторизуется через Google OAuth И профиль загружен, ТО приложение ДОЛЖНО автоматически перенаправить пользователя на Agents (главный экран приложения)

1.8. КОГДА происходит ошибка авторизации (обмен токенов ИЛИ загрузка профиля), ТО приложение ДОЛЖНО показать LoginError компонент с описанием ошибки

1.9. КОГДА пользователь выходит из системы (logout), ТО приложение ДОЛЖНО очистить все токены авторизации (см. google-oauth-auth.7) И автоматически перенаправить пользователя на экран логина

#### Нефункциональные Требования

**Производительность:**
- Проверка статуса авторизации должна выполняться менее чем за 100ms
- Перенаправление между экранами должно происходить мгновенно (< 50ms)

**Безопасность:**
- Защищенные маршруты должны быть недоступны без валидной авторизации
- При ошибке проверки авторизации система должна считать пользователя неавторизованным (fail-safe)

**Надежность:**
- Система навигации должна корректно работать при ошибках IPC коммуникации
- При ошибках навигации должен использоваться fallback механизм

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Кастомная анимация переходов между экранами
- История навигации (back/forward)
- Deep linking в защищенные экраны
- Сохранение последнего посещенного экрана
