# Документ Требований: Real-time Events System

## Введение

Данный документ описывает требования к системе событий реального времени (Real-time Events System) для приложения Clerkly. Система обеспечивает синхронизацию данных между различными частями приложения и автоматическое обновление UI при изменениях.

## Глоссарий

- **Event (Событие)** - Уведомление о произошедшем действии, содержащее тип и данные
- **Producer (Продюсер)** - Компонент, который генерирует события
- **Consumer (Консюмер)** - Компонент, который подписывается на события и реагирует на них
- **Event Bus** - Центральный механизм для публикации и подписки на события
- **Event Type** - Тип события (например, "user.login", "agent.created")
- **Event Payload** - Данные, передаваемые вместе с событием
- **Subscription (Подписка)** - Регистрация консюмера на получение событий определенного типа
- **IPC** - Inter-Process Communication, механизм взаимодействия между main и renderer процессами Electron
- **mitt** - Минималистичная библиотека event emitter (~200 bytes)

## Архитектурный Принцип

Система построена на паттерне Publisher-Subscriber (Pub/Sub):
- Продюсеры генерируют события
- Консюмеры подписываются на интересующие их типы событий
- Event Bus обеспечивает асинхронную доставку событий от продюсеров к консюмерам
- События передаются через IPC между main и renderer процессами
- События также распространяются внутри одного процесса (main или renderer)
- Вся коммуникация асинхронная

### Fire-and-Forget семантика

- Продюсер публикует событие и НЕ ждёт его обработки консюмерами
- Консюмеры обрабатывают события независимо друг от друга
- Async обработчики поддерживаются, но Event Bus не ждёт их завершения
- Порядок вызова обработчиков не гарантируется

## Требования

### Требование 1: Публикация событий

**ID:** realtime-events.1

**User Story:** Как продюсер, я хочу публиковать события, чтобы другие части приложения могли на них реагировать.

**Зависимости:** clerkly.3 (Logger)

#### Критерии Приемки

1.1. Продюсер ДОЛЖЕН иметь возможность опубликовать событие с указанием типа и данных

1.2. Событие ДОЛЖНО содержать:
   - `type: string` - тип события (например, "user.login", "agent.created")
   - `payload: object` - данные события
   - `timestamp: number` - время генерации события (Unix timestamp в миллисекундах)
   - `source: string` - идентификатор источника события (main/renderer)

1.3. Публикация события ДОЛЖНА быть асинхронной и не блокировать основной поток

1.4. События ДОЛЖНЫ передаваться через IPC между main и renderer процессами

1.5. События ДОЛЖНЫ распространяться внутри процесса (main→main, renderer→renderer)

1.6. Продюсер МОЖЕТ указать, нужно ли пересылать событие через IPC или только локально

**Тестируемость:** Да - через модульные тесты EventBus, функциональные тесты IPC

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should deliver event from main to renderer via IPC"
- `tests/functional/realtime-events.spec.ts` - "should deliver event from renderer to main via IPC"

### Требование 2: Подписка на события

**ID:** realtime-events.2

**User Story:** Как консюмер, я хочу подписаться на события определенного типа, чтобы реагировать на изменения данных.

**Зависимости:** Нет

#### Критерии Приемки

2.1. Консюмер ДОЛЖЕН иметь возможность подписаться на события по типу

2.2. Консюмер ДОЛЖЕН иметь возможность подписаться на несколько типов событий одновременно

2.3. Консюмер ДОЛЖЕН иметь возможность подписаться на все события (wildcard "*")

2.4. Подписка ДОЛЖНА возвращать функцию для отписки

2.5. При получении события консюмер ДОЛЖЕН получить callback с данными события

2.6. Callback ДОЛЖЕН выполняться асинхронно

2.7. ЕСЛИ callback выбрасывает исключение, ТО оно НЕ ДОЛЖНО влиять на других консюмеров

2.8. Консюмер ДОЛЖЕН автоматически отписываться при unmount компонента (для React)

2.9. Консюмер в renderer ДОЛЖЕН получать события из main процесса через IPC

2.10. Консюмер в main ДОЛЖЕН получать события из renderer процесса через IPC

**Тестируемость:** Да - через модульные тесты EventBus и React hooks

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should receive events across IPC boundary"

### Требование 3: Стандартные типы событий

**ID:** realtime-events.3

**User Story:** Как разработчик, я хочу иметь стандартизированные типы событий, чтобы обеспечить консистентность системы.

**Зависимости:** Нет

#### Критерии Приемки

3.1. Система ДОЛЖНА поддерживать следующие паттерны типов событий:
   - `{entity}.created` - создана новая сущность
   - `{entity}.updated` - обновлены данные сущности
   - `{entity}.deleted` - сущность удалена

3.2. Событие `{entity}.created` ДОЛЖНО содержать полные данные новой сущности

3.3. Событие `{entity}.updated` ДОЛЖНО содержать:
   - Полные данные сущности после обновления
   - Список измененных полей (`changedFields: string[]`) - ОБЯЗАТЕЛЬНО

3.4. Событие `{entity}.deleted` ДОЛЖНО содержать только ID удаленной сущности

3.5. Система ДОЛЖНА позволять определять кастомные типы событий

3.6. Все события ДОЛЖНЫ содержать timestamp для определения порядка событий

**Примечание:** Полный актуальный список типов событий находится в `src/shared/events/types.ts` — это единый источник истины.

**Тестируемость:** Да - через модульные тесты типов событий

### Требование 4: IPC интеграция

**ID:** realtime-events.4

**User Story:** Как разработчик, я хочу, чтобы события автоматически передавались между main и renderer процессами.

**Зависимости:** clerkly.1 (Electron IPC)

#### Критерии Приемки

4.1. EventBus в main процессе ДОЛЖЕН автоматически пересылать события в renderer через IPC

4.2. EventBus в renderer процессе ДОЛЖЕН автоматически пересылать события в main через IPC

4.3. События НЕ ДОЛЖНЫ дублироваться при пересылке через IPC

4.4. ЕСЛИ IPC канал недоступен, ТО события ДОЛЖНЫ доставляться только локально

4.5. Система ДОЛЖНА использовать существующие IPC каналы приложения

4.6. События ДОЛЖНЫ сериализоваться/десериализоваться корректно при передаче через IPC

4.7. IPC listeners в renderer ДОЛЖНЫ корректно очищаться при пересоздании EventBus (hot reload)

**Тестируемость:** Да - через функциональные тесты IPC

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should forward events from main to renderer"
- `tests/functional/realtime-events.spec.ts` - "should forward events from renderer to main"

### Требование 5: Обработка событий в UI

**ID:** realtime-events.5

**User Story:** Как пользователь, я хочу видеть изменения данных в UI без необходимости обновлять страницу.

**Зависимости:** Нет

#### Критерии Приемки

5.1. КОГДА приходит событие `{entity}.created`, ТО UI ДОЛЖЕН обновиться для отображения новой сущности

5.2. КОГДА приходит событие `{entity}.updated`, ТО данные сущности в UI ДОЛЖНЫ обновиться

5.3. КОГДА приходит событие `{entity}.deleted`, ТО сущность ДОЛЖНА быть удалена из UI

5.4. Обновления UI ДОЛЖНЫ происходить плавно без мерцания

5.5. События с более старым timestamp ДОЛЖНЫ игнорироваться, если уже обработано событие с более новым timestamp для той же сущности

**Тестируемость:** Да - через функциональные тесты UI

#### Функциональные Тесты

- `tests/functional/realtime-events.spec.ts` - "should update UI on entity.created"
- `tests/functional/realtime-events.spec.ts` - "should update UI on entity.updated"
- `tests/functional/realtime-events.spec.ts` - "should remove from UI on entity.deleted"

### Требование 6: Производительность

**ID:** realtime-events.6

**User Story:** Как разработчик, я хочу, чтобы система событий работала эффективно и не создавала проблем с производительностью.

**Зависимости:** Нет

#### Критерии Приемки

6.1. Обработка события ДОЛЖНА занимать < 10ms

6.2. Система ДОЛЖНА поддерживать минимум 100 событий в секунду без деградации производительности

6.3. События одного типа для одной сущности МОГУТ объединяться (batching) в пределах одного tick event loop

6.4. Старые неиспользуемые подписки ДОЛЖНЫ автоматически очищаться

6.5. Кэш timestamp'ов (lastEventTimestamps) ДОЛЖЕН очищаться:
   - При получении события `{entity}.deleted` — удалять запись для этой сущности
   - Периодически (раз в минуту) — удалять записи старше 6 часов

6.6. Порядок доставки событий для разных сущностей НЕ гарантируется

6.7. Размер payload события НЕ ДОЛЖЕН превышать 1MB (рекомендация: < 100KB)

**Тестируемость:** Да - через модульные и property-based тесты

### Требование 9: Снапшоты моделей в событиях

**ID:** realtime-events.9

**User Story:** Как разработчик, я хочу чтобы события содержали полные снапшоты моделей, чтобы UI мог обновляться без дополнительных запросов к БД.

**Зависимости:** Нет

#### Критерии Приемки

9.1. Каждое событие, связанное с моделью ORM, ДОЛЖНО содержать два обязательных поля:
   - `timestamp: number` - время генерации события (Unix timestamp в миллисекундах)
   - Ссылка на снапшот модели (например, `agent: AgentSnapshot`, `message: MessageSnapshot`)

9.2. Снапшот модели ДОЛЖЕН содержать:
   - Все поля модели ORM (минимум)
   - Опционально: вычисляемые поля (например, `status` для агента)

9.3. Снапшот модели ДОЛЖЕН быть отдельным интерфейсом (например, `AgentSnapshot`, `MessageSnapshot`)

9.4. Снапшот ДОЛЖЕН использовать типы, удобные для передачи через IPC:
   - Даты как `number` (Unix timestamp в миллисекундах) вместо `string` (ISO 8601)
   - JSON объекты распарсены (например, `payload: MessagePayload` вместо `payloadJson: string`)

9.5. Менеджер модели ДОЛЖЕН иметь приватный метод `toEventSnapshot()` для конвертации ORM модели в снапшот

9.6. Метод `toEventSnapshot()` ДОЛЖЕН:
   - Вычислять все вычисляемые поля (например, статус агента)
   - Парсить JSON поля в объекты
   - Конвертировать типы данных для IPC
   - Бросать ошибку если конвертация невозможна (не возвращать null)

9.7. События ДОЛЖНЫ генерироваться с полными снапшотами:
   - `AgentCreatedEvent` → `{ agent: AgentSnapshot, timestamp: number }`
   - `AgentUpdatedEvent` → `{ agent: AgentSnapshot, timestamp: number }`
   - `AgentArchivedEvent` → `{ agent: AgentSnapshot, timestamp: number }`
   - `MessageCreatedEvent` → `{ message: MessageSnapshot, timestamp: number }`
   - `MessageUpdatedEvent` → `{ message: MessageSnapshot, timestamp: number }`

9.8. UI ДОЛЖЕН использовать данные из снапшота напрямую без дополнительных запросов

**Примеры снапшотов:**

```typescript
// AgentSnapshot - содержит все поля Agent + вычисляемое поле status
interface AgentSnapshot {
  id: string;
  name: string;
  createdAt: number;  // Unix timestamp
  updatedAt: number;  // Unix timestamp
  archivedAt: number | null;  // Unix timestamp
  status: AgentStatus;  // Вычисляемое поле
}

// MessageSnapshot - содержит все поля Message + распарсенный payload
interface MessageSnapshot {
  id: number;
  agentId: string;
  timestamp: number;  // Unix timestamp
  payload: MessagePayload;  // Распарсенный JSON
}
```

**Тестируемость:** Да - через модульные тесты менеджеров и событий

## Нефункциональные Требования

### Надежность

- События НЕ ДОЛЖНЫ теряться при нормальной работе приложения
- ЕСЛИ консюмер недоступен, событие ДОЛЖНО быть пропущено (at-most-once delivery)
- Ошибки в одном консюмере НЕ ДОЛЖНЫ влиять на других консюмеров

### Инициализация

- MainEventBus ДОЛЖЕН инициализироваться при старте main процесса
- RendererEventBus ДОЛЖЕН инициализироваться при старте renderer процесса
- IPC handlers ДОЛЖНЫ регистрироваться до создания BrowserWindow

### Тестирование

- EventBus ДОЛЖЕН предоставлять метод `resetInstance()` для сброса singleton в тестах
- Метод `resetInstance()` ДОЛЖЕН вызывать `destroy()` перед сбросом

### Производительность

- Публикация события: < 1ms
- Доставка события консюмеру: < 10ms
- Поддержка минимум 100 событий/сек
- Память: < 10MB для Event Bus

### Масштабируемость

- Поддержка до 1000 активных подписок одновременно
- Поддержка до 100 типов событий

### Отладка

- Все события ДОЛЖНЫ логироваться с уровнем debug через Logger

## Зависимости

### Внешние библиотеки
- **mitt** (~200 bytes) - минималистичный event emitter для локальных событий внутри процесса
  - Функциональный API (методы не зависят от `this`)
  - Полная TypeScript типизация
  - Wildcard подписка (`*`)
  - API: `on(type, handler)`, `off(type, handler?)`, `emit(type, payload)`, `all.clear()`

### Внутренние компоненты
- `Logger` - для логирования событий и ошибок
- Electron IPC (ipcMain, ipcRenderer) - для передачи событий между main и renderer процессами
- Preload script + contextBridge - для безопасной передачи событий через IPC (contextIsolation: true)

## Ограничения

### Текущая реализация

- Нет персистентности событий (события существуют только в памяти)
- Нет гарантии доставки (at-most-once)
- Нет поддержки offline режима
- Нет replay событий
- Всегда одно окно (нет синхронизации между окнами)

### Будущие улучшения

- Event sourcing для полной истории изменений
- Персистентность событий в БД
- Replay событий для восстановления состояния

## Вне Области Применения

Следующие элементы явно исключены из данной спецификации:

- Синхронизация между вкладками/окнами браузера
- Синхронизация с сервером (только локальная синхронизация)
- Конфликт-резолюция при одновременном редактировании
- Транзакции и откат изменений
- Шифрование событий
- Аутентификация и авторизация событий
- Dashboard для метрик событий
- Throttling/debouncing на уровне EventBus
